4.0 Data Curation untuk Pre-training // kualitas data > kuantitas model

Sebelum melatih model bahasa, ada satu keputusan yang dampaknya melebihi semua hyperparameter: data apa yang digunakan untuk training? Kualitas data pre-training adalah prediktor tunggal terkuat dari performa model — jauh di atas jumlah parameter atau arsitektur. LLM terbaik saat ini (LLaMA 3, Claude, Gemini) berinvestasi besar-besaran dalam data curation pipeline yang ketat.

Data curation untuk LLM bukan sekadar "membersihkan teks". Ia melibatkan pipeline bertahap: deduplication (menghapus konten duplikat dan near-duplicate yang memboroskan compute dan menyebabkan memorization), quality filtering (menghapus teks berkualitas rendah menggunakan classifier atau heuristics), toxicity & PII filtering, dan data mixing (menyeimbangkan proporsi dari berbagai sumber: web, code, books, academic papers, multilingual).

Pipeline data curation — dari raw dump ke training-ready

  • Deduplication — Exact dedup (hash-based) menghapus duplikat sempurna. Near-dedup (MinHash, SimHash) menghapus dokumen yang 80%+ mirip — ini jauh lebih penting karena web penuh dengan konten boilerplate dan mirror. Dampak: mengurangi memorization, meningkatkan generalization, dan menghemat compute training.
  • Quality filtering — Gunakan lightweight classifier (fastText, KenLM perplexity) atau heuristic rules (panjang teks > 100 karakter, proporsi stop words, jumlah unique tokens). Pilihan: aggressive filtering menghasilkan data lebih kecil tapi lebih bersih vs permissive yang mempertahankan diversity.
  • Data mixing — Tentukan proporsi sumber data: web (Common Crawl, C4) 60–70%, code (GitHub, Stack Overflow) 10–15%, books/academic 5–10%, Wikipedia 3–5%, multilingual 5–10%. LLaMA paper menunjukkan bahwa menambah code dan academic text secara dramatis meningkatkan reasoning ability.
  • Decontamination — Hapus data training yang overlap dengan benchmark evaluation sets. Jika model "melihat" soal ujian saat training, evaluasi menjadi tidak valid. Ini tantangan besar karena benchmark dataset sering tersebar di web.
Pythondata_curation.py
import hashlib
from datasketch import MinHash, MinHashLSH
from collections import defaultdict
import re

class DataCurationPipeline:
    def __init__(self, min_text_length: int = 100,
                 near_dedup_threshold: float = 0.8):
        self.seen_hashes = set()
        self.lsh = MinHashLSH(threshold=near_dedup_threshold, num_perm=128)
        self.min_length = min_text_length

    def is_duplicate_exact(self, text: str) -> bool:
        text_hash = hashlib.sha256(text.encode()).hexdigest()
        if text_hash in self.seen_hashes: return True
        self.seen_hashes.add(text_hash)
        return False

    def is_duplicate_near(self, text: str) -> bool:
        # MinHash: representasi compressed dari set n-gram
        m = MinHash(num_perm=128)
        for word in text.lower().split():
            m.update(word.encode('utf8'))
        result = self.lsh.query(m)
        self.lsh.insert(hash(text), m)
        return len(result) > 0

    def quality_filter(self, text: str) -> tuple[bool, dict]:
        metrics = {}
        # Length check
        if len(text) < self.min_length:
            return False, {"reason": "too_short"}
        # Word count must be reasonable
        words = text.split()
        if len(words) < 10:
            return False, {"reason": "too_few_words"}
        # Mean word length — filter gibberish
        avg_word_len = sum(len(w) for w in words) / len(words)
        if avg_word_len < 2.5 or avg_word_len > 12:
            return False, {"reason": "abnormal_word_length"}
        # Ratio of "normal" ASCII text — filter non-text content
        ascii_ratio = sum(1 for c in text if 32 <= ord(c) <= 126) / len(text)
        if ascii_ratio < 0.7:
            return False, {"reason": "non_text_content"}
        return True, {"avg_word_len": avg_word_len, "words": len(words)}

# === Data mixing strategy ===
def sample_mixture(sources: dict[str, list], ratios: dict[str, float],
                    total: int) -> list:
    """Sample dari berbagai sumber sesuai rasio mixing."""
    import random
    sampled = []
    for source, docs in sources.items():
        n = max(1, int(total * ratios.get(source, 0.1)))
        sampled.extend(random.sample(docs, min(n, len(docs))))
    random.shuffle(sampled)
    return sampled[:total]

4.1 Paradigma Pre-training // kenapa self-supervised learning revolusioner

Sebelum era pre-training, setiap NLP task membutuhkan model yang dilatih dari nol pada labeled data yang spesifik — mahal, lambat, dan tidak scalable. Pre-training membalikkan paradigma ini: latih satu model besar dengan self-supervised objective pada corpus teks yang sangat besar (tidak butuh label), lalu adapkan ke task spesifik dengan sedikit data berlabel. Hasilnya: performa yang jauh lebih baik dengan jauh lebih sedikit labeled data.

Kunci keberhasilan pre-training adalah self-supervised learning: kita bisa membuat label dari teks itu sendiri tanpa anotasi manusia. Prediksi kata berikutnya (causal LM) atau prediksi kata yang di-mask (masked LM) — keduanya menghasilkan jumlah training signal yang tidak terbatas dari internet. Ini memungkinkan training pada data triliunan token, skala yang tidak mungkin dicapai dengan supervised learning tradisional.

Paradigma Lama vs Baru — Transfer Learning di NLP
Paradigma Lama — Per-Task Training
Task A: Sentiment
  → 50K labeled examples
  → Train model A dari scratch
  → Performa terbatas

Task B: NER
  → 100K labeled examples
  → Train model B dari scratch
  → Tidak bisa transfer ke Task A

Biaya: N × (data labeling + training)
Paradigma Baru — Pre-train + Fine-tune
Pre-training (1×):
  → 100B+ token unlabeled text
  → Pelajari general language understanding
  → Simpan sebagai "foundation model"

Fine-tuning (N×):
  Task A: 5K examples → state-of-the-art
  Task B: 2K examples → state-of-the-art
  Task C: zero-shot → surprisingly good

Biaya: 1× besar + N× kecil

Dua objektif pre-training yang membentuk LLM modern

  • Causal Language Modeling (CLM) — prediksi token berikutnya dari konteks kiri. Cocok dengan decoder-only architecture. Tidak perlu masking — setiap token menggunakan semua token sebelumnya sebagai konteks. GPT, LLaMA, Claude semua menggunakan CLM.
  • Masked Language Modeling (MLM) — mask ~15% token secara acak, prediksi token yang di-mask dari konteks kedua arah. Cocok dengan encoder-only architecture. Menghasilkan representasi bidirectional yang kaya. BERT, RoBERTa menggunakan MLM.
  • Span Corruption (T5) — mask span token secara berurutan (bukan individual), decoder memprediksi span yang hilang. Lebih efisien dari MLM karena model hanya perlu menghasilkan token yang hilang, bukan merekonstruksi seluruh input.
Kenapa pre-training bisa transfer ke task lain?Selama pre-training, model dipaksa untuk mengerti konteks, syntax, semantik, fakta dunia, dan reasoning — karena semua informasi ini berguna untuk memprediksi token. Representasi yang dibangun tidak task-specific, melainkan merupakan model umum tentang bagaimana bahasa bekerja. Fine-tuning hanya perlu mengarahkan representasi ini ke task spesifik.
Pythonpretraining_objective.py
import torch
import torch.nn.functional as F
import random

def causal_lm_loss(model, batch_tokens: torch.Tensor) -> torch.Tensor:
    """CLM: prediksi token t+1 dari token 0..t"""
    # batch_tokens: [batch, seq_len]
    inputs = batch_tokens[:, :-1]   # [batch, seq-1] — semua kecuali token terakhir
    targets = batch_tokens[:, 1:]   # [batch, seq-1] — semua kecuali token pertama

    logits = model(inputs)           # [batch, seq-1, vocab_size]
    return F.cross_entropy(
        logits.reshape(-1, logits.size(-1)),  # [batch*(seq-1), vocab]
        targets.reshape(-1)                   # [batch*(seq-1)]
    )

def create_mlm_batch(tokens: torch.Tensor, mask_prob: float = 0.15,
                      mask_token_id: int = 103, vocab_size: int = 30522):
    """Buat masked version dari tokens untuk MLM pre-training (BERT style)."""
    labels = tokens.clone()
    masked = tokens.clone()

    # Mask probability per token
    prob_matrix = torch.full(tokens.shape, mask_prob)
    mask_indices = torch.bernoulli(prob_matrix).bool()

    for idx in mask_indices.nonzero():
        r = random.random()
        if r < 0.8:
            # 80%: ganti dengan [MASK] token
            masked[idx[0], idx[1]] = mask_token_id
        elif r < 0.9:
            # 10%: ganti dengan token random (mencegah model bergantung pada [MASK])
            masked[idx[0], idx[1]] = random.randint(0, vocab_size - 1)
        # 10%: biarkan unchanged (tapi masih hitung loss)

    # Hanya hitung loss pada posisi yang di-mask
    labels[~mask_indices] = -100  # CrossEntropy mengabaikan -100
    return masked, labels

4.2 BERT & Bidirectional Models // memahami konteks penuh

BERT (Bidirectional Encoder Representations from Transformers, Devlin et al., 2018) merevolusi NLP dengan satu insight sederhana: model bahasa yang baik harus memahami kata dalam konteks kedua arahnya, bukan hanya konteks kiri (seperti GPT) atau pemrosesan sekuensial (seperti LSTM). "Pengacara terbaik" berbeda maknanya dari "pengacara terbaik yang pernah dipecat" — BERT bisa menangkap nuansa ini karena melihat seluruh kalimat sekaligus.

BERT menggunakan dua pre-training objectives: Masked Language Modeling (MLM) yang memaksa model membangun representasi bidirectional, dan Next Sentence Prediction (NSP) yang mengajarkan hubungan antar kalimat. Meskipun NSP kemudian terbukti kurang penting (RoBERTa menghapus NSP tanpa penurunan performa), MLM tetap menjadi signature BERT dan varian-variannya.

Pythonbert_finetuning.py
from transformers import BertTokenizer, BertForSequenceClassification
from transformers import TrainingArguments, Trainer
from datasets import load_dataset
import torch
import numpy as np

# === Load pre-trained BERT + classification head ===
model_name = "bert-base-uncased"
tokenizer = BertTokenizer.from_pretrained(model_name)
model = BertForSequenceClassification.from_pretrained(
    model_name,
    num_labels=2  # positif / negatif
)

# === Tokenisasi dataset ===
dataset = load_dataset("imdb")

def tokenize_function(examples):
    return tokenizer(
        examples["text"],
        truncation=True,
        max_length=512,    # BERT max context
        padding="max_length"
    )

tokenized = dataset.map(tokenize_function, batched=True)

# === Fine-tuning dengan HuggingFace Trainer ===
training_args = TrainingArguments(
    output_dir="./bert-sentiment",
    num_train_epochs=3,
    per_device_train_batch_size=16,
    warmup_ratio=0.1,       # 10% steps untuk warmup
    learning_rate=2e-5,      # BERT fine-tuning: 1e-5 sampai 5e-5
    weight_decay=0.01,
    evaluation_strategy="epoch",
    save_strategy="epoch",
    load_best_model_at_end=True,
    fp16=True,               # mixed precision — 2× lebih cepat di modern GPU
)

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=tokenized["train"],
    eval_dataset=tokenized["test"],
    compute_metrics=lambda p: {
        "accuracy": (np.argmax(p.predictions, axis=1) == p.label_ids).mean()
    }
)

trainer.train()

# === Manual inference ===
inputs = tokenizer("This movie was absolutely fantastic!",
                   return_tensors="pt", truncation=True)
with torch.no_grad():
    logits = model(**inputs).logits
    pred = logits.argmax()
    print(f"Sentiment: {'POSITIVE' if pred == 1 else 'NEGATIVE'}")

BERT variants yang perlu diketahui

  • RoBERTa — BERT yang dilatih lebih lama, dengan data lebih banyak, tanpa NSP, dan dynamic masking. Secara konsisten outperform BERT. Gunakan RoBERTa sebagai default jika kamu butuh encoder-only model.
  • DistilBERT — BERT yang di-distill menjadi 40% lebih kecil, 60% lebih cepat, dengan hanya 3% penurunan performa. Ideal untuk deployment dengan resource terbatas.
  • DeBERTa — BERT dengan disentangled attention (posisi dan konten dipisahkan). State-of-the-art untuk banyak understanding benchmarks hingga 2022.
  • ELECTRA — pendekatan berbeda: alih-alih MLM, gunakan replaced token detection (biner: apakah token ini yang asli atau diganti oleh generator kecil?). Lebih efisien dari BERT karena menghitung loss pada semua token, bukan hanya yang di-mask.
Learning rate untuk fine-tuning BERTGunakan learning rate kecil: 1e-5 hingga 5e-5. Learning rate yang terlalu besar menyebabkan "catastrophic forgetting" — model melupakan representasi yang dipelajari saat pre-training. Selalu gunakan warmup (5–10% dari total steps) untuk warming up optimizer state sebelum full learning rate. Layer-wise learning rate decay (LLRD) — layer atas belajar lebih cepat dari layer bawah — bisa meningkatkan performa fine-tuning secara konsisten.

4.3 GPT & Autoregressive Models // in-context learning & few-shot

GPT (Generative Pre-trained Transformer) mengambil jalur yang berbeda dari BERT: decoder-only architecture dengan causal language modeling. Setiap token hanya bisa memperhatikan token sebelumnya — ini membuatnya natural untuk text generation, tapi mungkin kurang optimal untuk understanding dibanding BERT's bidirectional attention. Atau begitulah anggapan awal, sebelum GPT-3 dan keluarganya membuktikan bahwa dengan skala yang cukup besar, model autoregresif bisa mengungguli BERT bahkan di understanding tasks.

Salah satu emergent capability yang paling mengejutkan dari model autoregresif besar adalah in-context learning: kemampuan model untuk "belajar" dari contoh-contoh yang diberikan dalam prompt, tanpa update weight apapun. GPT-3 (Brown et al., 2020) menunjukkan bahwa dengan memberikan beberapa contoh input-output (few-shot), model bisa melakukan task baru dengan akurasi yang mengagumkan — sebuah bentuk learning yang tidak ada analoginya di ML tradisional.

Pythongpt_inference.py
from transformers import GPT2LMHeadModel, GPT2Tokenizer, pipeline
import torch

# === GPT-2 text generation ===
model = GPT2LMHeadModel.from_pretrained("gpt2")
tokenizer = GPT2Tokenizer.from_pretrained("gpt2")

prompt = "The future of artificial intelligence"
inputs = tokenizer(prompt, return_tensors="pt")

# Berbagai decoding strategies
# 1. Greedy decoding — deterministic, tapi sering repetitif
greedy = model.generate(**inputs, max_new_tokens=50)

# 2. Beam search — better quality, lebih lambat
beam = model.generate(**inputs, max_new_tokens=50, num_beams=5, early_stopping=True)

# 3. Sampling dengan temperature + top-k + top-p (nucleus sampling)
# Ini adalah decoding standard untuk conversational LLM
sampled = model.generate(
    **inputs,
    max_new_tokens=100,
    do_sample=True,
    temperature=0.8,    # <1 = lebih fokus, >1 = lebih random
    top_k=50,           # hanya dari 50 token dengan prob tertinggi
    top_p=0.92,         # nucleus: pilih dari token yang jumlah prob-nya < 0.92
    repetition_penalty=1.3,  # penalti untuk token yang sudah muncul
)

for output, name in [(greedy, "Greedy"), (beam, "Beam"), (sampled, "Sampled")]:
    text = tokenizer.decode(output[0], skip_special_tokens=True)
    print(f"\n[{name}]:\n{text}")

4.4 Fine-tuning Strategies // dari full fine-tuning hingga LoRA

Full fine-tuning — update semua parameter model untuk task spesifik — menghasilkan performa terbaik tapi sangat mahal: model 7B parameter membutuhkan sekitar 28GB GPU memory hanya untuk weights, belum termasuk gradients dan optimizer state. Ini mendorong lahirnya Parameter-Efficient Fine-tuning (PEFT): teknik yang mengupdate sebagian kecil parameter sambil mempertahankan sebagian besar model frozen, dengan performa yang mendekati full fine-tuning.

LoRA (Low-Rank Adaptation) adalah teknik PEFT yang paling populer saat ini. Ide dasarnya: weight update yang terjadi saat fine-tuning sebenarnya memiliki rank yang rendah (low intrinsic dimensionality). Alih-alih mempelajari matrix update ΔW berukuran penuh, LoRA mempelajari dekomposisi rank-r: ΔW = A·B, di mana A ∈ ℝ^(d×r) dan B ∈ ℝ^(r×k) dengan r << min(d, k). Hasilnya: hanya 0.1–1% parameter yang ditraining, tapi performa mendekati full fine-tuning.

Pythonlora_finetuning.py
from peft import LoraConfig, get_peft_model, TaskType, PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments
from trl import SFTTrainer
import torch

# === Setup LoRA config ===
lora_config = LoraConfig(
    r=16,               # rank — lebih tinggi = lebih banyak params, lebih powerful
    lora_alpha=32,      # scaling factor: alpha/r menentukan magnitude update
    target_modules=[    # layer mana yang di-adapt — biasanya attention projections
        "q_proj", "v_proj",  # minimal: query dan value
        "k_proj", "o_proj",  # optional: key dan output
        "gate_proj", "up_proj", "down_proj"  # FFN layers (opsional)
    ],
    lora_dropout=0.1,
    bias="none",
    task_type=TaskType.CAUSAL_LM,
)

# === Load model dan aplikasikan LoRA ===
base_model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.1-8B",
    torch_dtype=torch.bfloat16,   # bf16: 16-bit dengan range seperti fp32
    device_map="auto",           # otomatis distribusi ke GPU yang tersedia
)
model = get_peft_model(base_model, lora_config)
model.print_trainable_parameters()
# trainable params: 6,815,744 || all params: 8,036,614,144 || trainable%: 0.085%

# === SFT (Supervised Fine-tuning) Training ===

Full Fine-tuning

Semua parameter diupdate. Performa terbaik. Membutuhkan GPU memory besar (model besar membutuhkan A100 80GB atau lebih). Hasilkan model terpisah per task. Cocok jika dataset besar (>100K) dan compute tersedia.

LoRA

Hanya 0.1–1% parameter ditraining. GPU memory jauh lebih hemat. Bisa dijalankan di GPU consumer (RTX 3090). Performa mendekati full FT. Adapter bisa di-swap tanpa reload base model. Pilihan default untuk most use cases.

QLoRA

LoRA + quantization (int4). Base model di-quantize ke 4-bit, LoRA adapters tetap float16. Bisa fine-tune model 65B di single A100 80GB. Sedikit performa penalty dari quantization, tapi trade-off yang sangat baik.

Prompt Tuning

Hanya melatih soft prompt (virtual tokens) di awal input. Parameter minimal (ribuan, bukan jutaan). Performa baik untuk model besar, tapi buruk untuk model kecil. Cocok untuk very resource-constrained deployment.

Memilih rank LoRA yang tepatRank r mengontrol kapasitas adapter. Mulai dengan r=8 atau r=16 untuk tugas sederhana. Untuk task domain-specific yang kompleks, r=32 atau r=64. Lebih tinggi dari r=128 jarang memberikan improvement yang sebanding. Selalu mulai kecil dan naikkan jika validasi loss menunjukkan underfitting.

4.5 RLHF & Alignment // melatih model agar bermanfaat dan aman

Sebuah model yang sangat baik dalam memprediksi token berikutnya belum tentu berguna atau aman. Model yang dilatih hanya dengan causal LM akan dengan senang hati menghasilkan konten berbahaya, jawaban yang menyesatkan, atau teks yang tidak sesuai instruksi pengguna — karena itu semua mungkin muncul dalam data training. Alignment adalah proses membuat model berperilaku sesuai dengan nilai dan preferensi manusia.

RLHF (Reinforcement Learning from Human Feedback) adalah teknik alignment yang pertama kali berhasil di skala besar, digunakan untuk melatih InstructGPT dan ChatGPT. Prosesnya tiga tahap: (1) Supervised Fine-Tuning (SFT) pada demonstrasi respons berkualitas tinggi, (2) melatih Reward Model yang memprediksi preferensi manusia dari pasangan respons yang diberi ranking, dan (3) mengoptimasi policy model menggunakan Proximal Policy Optimization (PPO) dengan reward model sebagai sinyal.

Pipeline RLHF — Tiga Tahap Training
Stage 1: Supervised Fine-Tuning (SFT)
Fine-tune base LLM pada demonstrasi instruksi berkualitas tinggi yang ditulis labeler
Stage 2: Reward Model Training
Labeler memberi ranking dua respons → train model yang memprediksi ranking ini
Stage 3: RL Fine-tuning dengan PPO
Gunakan reward model sebagai "judge" — optimasi SFT model agar mendapat reward tinggi
Aligned Model (InstructGPT, ChatGPT style)
Lebih helpful, harmless, dan honest dari base LLM

Alternatif modern untuk RLHF

  • DPO (Direct Preference Optimization) — menghilangkan reward model eksplisit dan PPO. Langsung optimasi LLM dari preference pairs menggunakan supervised-style loss. Lebih stabil, lebih simpel, performa seringkali match atau overtake RLHF. Standar de facto 2024.
  • Constitutional AI (Anthropic) — model mengkritik responsnya sendiri berdasarkan serangkaian prinsip (konstitusi), lalu merevisinya. Mengurangi kebutuhan human labeling. Hasilkan model yang lebih consistent dengan prinsip yang ditetapkan.
  • RLAIF (RL from AI Feedback) — ganti human labeler dengan LLM yang lebih kuat sebagai penilai. Scalable tapi rawan bias dari model penilai.
  • Reward Model sebagai verifier — selain training, reward model digunakan saat inference untuk best-of-N sampling: generate N respons, pilih yang mendapat reward tertinggi. Simple tapi efektif.
Reward hacking dan alignment taxModel yang dioptimasi untuk reward model bisa mempelajari cara "mengeksploitasi" reward model — menghasilkan respons yang terlihat bagus di mata reward model tapi tidak benar-benar berkualitas (Goodhart's Law). RLHF juga memiliki "alignment tax": model aligned seringkali sedikit lebih lemah di benchmark faktual murni karena dioptimasi untuk helpfulness yang luas. Ini adalah area riset terbuka yang sangat aktif di 2024–2025.

4.6 LLM Evaluation // bagaimana tahu model mana yang benar-benar lebih baik?

Mengevaluasi LLM adalah salah satu masalah tersulit dalam AI modern. Tidak seperti model ML klasik yang punya ground truth yang jelas (fraud/not fraud, churn/not churn), kualitas teks bersifat subjektif, multi-dimensional, dan konteks-dependent. Model yang menang di benchmark akademik seringkali bukan yang terbaik untuk use case spesifik kamu.

Ada tiga pendekatan utama yang saling melengkapi: benchmark evaluation (MMLU, HellaSwag, GSM8K, HumanEval — terstandarisasi tapi rentan contamination), human evaluation (LMSys Chatbot Arena, pairwise comparison — gold standard tapi mahal), dan LLM-as-judge (menggunakan LLM yang lebih kuat untuk menilai output LLM lain — scalable tapi punya bias sendiri). Tidak ada yang sempurna — kombinasi ketiganya memberikan gambaran paling akurat.

Tiga pilar LLM evaluation

  • Benchmark Evaluation — Dataset terstandarisasi dengan jawaban benar. MMLU (57 subjek, multiple choice), GSM8K (math reasoning), HumanEval (code generation), HellaSwag (commonsense). Masalah utama: benchmark contamination — data training tidak sengaja mengandung soal ujian. Solusi: gunakan benchmark yang frequently updated (LiveBench), private test sets, atau dynamic generation.
  • Human Evaluation — LMSys Chatbot Arena: user bandingkan dua model anonim, voting mana yang lebih baik. Hasilnya diranking dengan Elo rating system (seperti catur). Keunggulan: menangkap "usefulness" yang tidak bisa diukur benchmark. Kelemahan: mahal, lambat, dan ada positional bias (model yang muncul di kiri cenderung dipilih).
  • LLM-as-Judge — GPT-4 atau Claude digunakan untuk menilai output LLM lain. Contoh: beri dua respons, minta judge memilih yang lebih baik dengan justifikasi. MT-Bench dan AlpacaEval adalah contoh benchmark yang menggunakan pendekatan ini. Keunggulan: scalable, cepat, reproducible. Kelemahan: judge model punya bias sendiri (position bias, verbosity bias — cenderung memilih respons yang lebih panjang).

import numpy as np
from scipy import stats

class EloRating:
    """Elo rating system untuk pairwise model comparison (Arena-style)."""
    def __init__(self, k_factor: int = 32, base_rating: int = 1000):
        self.ratings = {}
        self.k = k_factor
        self.base = base_rating

    def get_rating(self, model: str) -> float:
        return self.ratings.get(model, self.base)

    def expected_score(self, rating_a: float, rating_b: float) -> float:
        return 1 / (1 + 10 ** ((rating_b - rating_a) / 400))

    def update(self, winner: str, loser: str):
        rw = self.get_rating(winner)
        rl = self.get_rating(loser)
        expected = self.expected_score(rw, rl)
        self.ratings[winner] = rw + self.k * (1 - expected)
        self.ratings[loser]  = rl + self.k * (0 - (1 - expected))

# === Contoh: simulasi Arena comparison ===
elo = EloRating()
# Model A menang 7 dari 10 kali melawan Model B
for _ in range(7): elo.update("Model-A", "Model-B")
for _ in range(3): elo.update("Model-B", "Model-A")
print(f"Model A ELO: {elo.get_rating('Model-A'):.0f}")
print(f"Model B ELO: {elo.get_rating('Model-B'):.0f}")

# === Statistical significance untuk pairwise comparison ===
def bootstrap_confidence(model_a_wins: int, total_games: int,
                          n_bootstrap: int = 10000) -> tuple[float, float]:
    observed_winrate = model_a_wins / total_games
    bootstraps = np.random.binomial(total_games, observed_winrate, n_bootstrap) / total_games
    ci_low = np.percentile(bootstraps, 2.5)
    ci_high = np.percentile(bootstraps, 97.5)
    return ci_low, ci_high
