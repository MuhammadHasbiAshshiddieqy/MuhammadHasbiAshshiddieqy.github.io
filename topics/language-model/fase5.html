5.1 Scaling Laws // prediksi kemampuan dari compute

Scaling laws adalah salah satu penemuan paling mengejutkan dan praktis dalam riset LLM: performa model (diukur sebagai loss) mengikuti power law yang sangat predictable terhadap tiga variabel — jumlah parameter model (N), jumlah token training (D), dan compute yang tersedia (C). Artinya, sebelum melatih model besar yang mahal, kita bisa dengan akurat memprediksi berapa loss yang akan dicapai.

Paper Chinchilla (Hoffmann et al., 2022) merevolusi pemahaman kita tentang compute-optimal training: untuk budget compute tertentu, kita tidak harus melatih model sebesar mungkin — kita harus menyeimbangkan ukuran model dengan jumlah data. Temuan Chinchilla: model optimal harus dilatih dengan sekitar 20 token per parameter. GPT-3 (175B params) seharusnya dilatih dengan 3.5T token, bukan 300B yang digunakan — model sebesar GPT-3 "underfit" karena kurang data.

Tiga konsep penting dari scaling literature

  • Compute-optimal training (Chinchilla) — N tokens ≈ 20 × N params adalah sweet spot. LLaMA 7B (LLaMA 2) dilatih dengan 2T token (285× dari Chinchilla-optimal) — justru overtraining, tapi hasilnya model yang lebih baik saat inference dengan biaya yang sama. Ini menunjukkan bahwa "optimal untuk training cost" berbeda dari "optimal untuk inference deployment".
  • Emergent abilities — kemampuan yang tidak ada di model kecil tiba-tiba muncul di model skala tertentu tanpa transisi gradual: few-shot arithmetic, chain-of-thought reasoning, multilingual understanding. Apakah ini benar-benar emergent atau hanya artifact dari metrik evaluasi yang kasar masih diperdebatkan.
  • Inference-optimal vs training-optimal — trade-off nyata: kamu bisa melatih model kecil lebih lama (lebih murah saat deployment) vs model besar lebih singkat (lebih murah saat training). Pilihan bergantung pada volume inference yang diharapkan.
Scaling laws tidak mempredikti semua kemampuanScaling laws sangat akurat untuk prediksi loss, tapi kurang akurat untuk kemampuan spesifik. Model dengan loss identik bisa berbeda drastis dalam coding ability, mathematical reasoning, atau instruction following. RLHF dan fine-tuning juga mengubah kemampuan model secara dramatis tanpa mengubah base loss — ini tidak tercakup dalam scaling law klasik.
Pythonscaling_laws.py
import numpy as np
import matplotlib.pyplot as plt

# Chinchilla scaling law: L(N, D) ≈ E + A/N^alpha + B/D^beta
# E = irreducible entropy (~1.69 untuk teks Inggris)
# A, B, alpha, beta = konstanta fit dari data empiris

E = 1.69   # irreducible loss
A = 406.4
B = 410.7
alpha = 0.34
beta = 0.28

def chinchilla_loss(N: float, D: float) -> float:
    """Prediksi loss untuk N params dan D tokens."""
    return E + A / N**alpha + B / D**beta

def optimal_allocation(compute_budget: float, tokens_per_param: float = 20):
    """Hitung optimal N dan D untuk budget compute C = 6ND (approx)."""
    # C = 6 × N × D (rule of thumb: 6 FLOPs per token per parameter)
    # Chinchilla: D/N ≈ 20 (train 20 tokens per parameter)
    N_optimal = 0.5 * (compute_budget / (6 * tokens_per_param)) ** 0.5
    D_optimal = 20 * N_optimal
    return N_optimal, D_optimal

# Contoh: untuk $1M GPU budget (~10^23 FLOPs)
N, D = optimal_allocation(1e23)
print(f"Optimal model size: {N/1e9:.1f}B parameters")
print(f"Optimal training tokens: {D/1e9:.0f}B tokens")
predicted_loss = chinchilla_loss(N, D)
print(f"Predicted loss: {predicted_loss:.3f}")

# Visualisasi loss landscape sebagai heatmap N vs D
N_range = np.logspace(8, 12, 50)  # 100M to 1T params
D_range = np.logspace(10, 13, 50)  # 10B to 10T tokens
N_grid, D_grid = np.meshgrid(N_range, D_range)
loss_grid = chinchilla_loss(N_grid, D_grid)

5.2 Mixture of Experts // skala tanpa compute proporsional

Dense model — model Transformer standar — mengaktifkan semua parameter untuk setiap token yang diproses. Mixture of Experts (MoE) menawarkan alternatif yang lebih efisien: alih-alih satu FFN besar yang digunakan untuk semua token, miliki N "expert" FFN yang spesialis, dan hanya aktifkan K expert (biasanya 2) untuk setiap token. Hasilnya: model dengan total parameter sangat besar (parameter counts yang diklaim biasanya mengacu pada total parameter) tapi compute per token jauh lebih rendah.

Gemini 1.0 MoE, GPT-4 (spekulatif), Mixtral 8×7B, dan Grok adalah contoh nyata. Mixtral 8×7B memiliki total 47B parameter tapi hanya menggunakan ~13B parameter aktif per token — performa setara dengan model 65B dense tapi dengan biaya inference setara model 13B. Ini adalah trade-off yang sangat menarik untuk production deployment.

Pythonmixture_of_experts.py
import torch
import torch.nn as nn
import torch.nn.functional as F

class Expert(nn.Module):
    """Satu expert FFN — identik strukturnya, berbeda weightsnya."""
    def __init__(self, d_model: int, d_ff: int):
        super().__init__()
        self.fc1 = nn.Linear(d_model, d_ff)
        self.fc2 = nn.Linear(d_ff, d_model)

    def forward(self, x):
        return self.fc2(F.silu(self.fc1(x)))  # SiLU/Swish activation

class SparseMoE(nn.Module):
    """Sparse Mixture of Experts layer — pengganti FFN dalam Transformer."""
    def __init__(self, d_model: int, d_ff: int,
                 num_experts: int = 8, top_k: int = 2):
        super().__init__()
        self.num_experts = num_experts
        self.top_k = top_k

        # Router: memutuskan expert mana yang menangani token ini
        self.router = nn.Linear(d_model, num_experts, bias=False)

        # Pool of experts
        self.experts = nn.ModuleList([
            Expert(d_model, d_ff) for _ in range(num_experts)
        ])

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # x: [batch, seq_len, d_model]
        B, T, D = x.shape
        x_flat = x.view(-1, D)  # [B*T, d_model]

        # Router logits → select top-k experts per token
        router_logits = self.router(x_flat)           # [B*T, num_experts]
        router_weights = F.softmax(router_logits, dim=-1)
        top_k_weights, top_k_indices = torch.topk(router_weights, self.top_k, dim=-1)
        top_k_weights = top_k_weights / top_k_weights.sum(dim=-1, keepdim=True)  # normalize

        # Proses setiap token dengan top-k experts dan weighted sum hasilnya
        output = torch.zeros_like(x_flat)
        for k in range(self.top_k):
            expert_idx = top_k_indices[:, k]   # [B*T] — expert index per token
            weight = top_k_weights[:, k:k+1]  # [B*T, 1]
            for i, expert in enumerate(self.experts):
                mask = (expert_idx == i)       # token yang routing ke expert i
                if mask.any():
                    output[mask] += weight[mask] * expert(x_flat[mask])

        # Auxiliary load balancing loss — mencegah semua token ke expert sama
        # router_probs: rata-rata fraksi token per expert → harusnya uniform = 1/num_experts
        router_probs = router_weights.mean(dim=0)  # [num_experts]
        load_balance_loss = self.num_experts * (router_probs ** 2).sum()

        return output.view(B, T, D), load_balance_loss
Load balancing adalah critical untuk MoETanpa load balancing loss, router akan mengkolaps ke beberapa expert populer (expert collapse) — model effectively menggunakan 2 expert dari 8, kehilangan keunggulan MoE. Auxiliary loss memaksa distribusi token yang lebih merata. Di production, monitoring per-expert utilization adalah essential untuk mendeteksi masalah ini.

5.3 Inference Optimization // dari riset ke production

LLM inference adalah bottleneck production yang unik: tidak seperti model ML lain yang compute-bound, LLM inference seringkali memory bandwidth-bound — bottleneck bukan komputasi FLOPs, melainkan seberapa cepat kita bisa memindahkan weight dari memory ke compute units. Teknik inference optimization modern berfokus pada mengurangi data movement, meningkatkan hardware utilization, dan mengurangi compute aktual tanpa mengorbankan kualitas output.

KV Cache adalah optimasi paling fundamental: self-attention membutuhkan key dan value dari semua token sebelumnya. Tanpa caching, setiap token baru perlu recompute K dan V untuk semua token sebelumnya — O(n²) per step. Dengan KV cache, kita simpan K dan V yang sudah dihitung dan hanya hitung untuk token baru — O(n) per step. Di model besar dengan context window panjang, ukuran KV cache bisa melebihi ukuran model weight sendiri.

Pythoninference_optimization.py
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig

# === 1. Quantization (INT8) — 2× memory saving ===
bnb_config_int8 = BitsAndBytesConfig(
    load_in_8bit=True,
    llm_int8_threshold=6.0,   # outlier threshold — turunkan jika ada degradasi
)

# === 2. INT4 Quantization (QLoRA/GPTQ style) — 4× memory saving ===
bnb_config_int4 = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",      # NF4: normal float 4 — better than standard int4
    bnb_4bit_use_double_quant=True,  # quantize quantization constants juga
    bnb_4bit_compute_dtype=torch.bfloat16,
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3.1-8B-Instruct",
    quantization_config=bnb_config_int4,
    device_map="auto",
)
# Memory: ~8GB float16 → ~4GB int8 → ~2.5GB int4

# === 3. KV Cache Memory Estimation ===
def kv_cache_size_bytes(
    num_layers: int,
    num_heads: int,
    head_dim: int,
    max_seq_len: int,
    batch_size: int,
    dtype_bytes: int = 2  # float16 = 2 bytes
) -> float:
    """Hitung ukuran KV cache dalam GB."""
    # 2 × (K + V) × layers × heads × head_dim × seq × batch
    total = 2 * num_layers * num_heads * head_dim * max_seq_len * batch_size * dtype_bytes
    return total / (1024**3)  # convert ke GB

# LLaMA 3.1 8B: 32 layers, 32 heads, 128 head_dim
kv_gb = kv_cache_size_bytes(32, 32, 128, max_seq_len=8192, batch_size=1)
print(f"KV cache (8K context, bs=1): {kv_gb:.2f} GB")  # ~4 GB
kv_gb_128k = kv_cache_size_bytes(32, 32, 128, max_seq_len=131072, batch_size=1)
print(f"KV cache (128K context, bs=1): {kv_gb_128k:.2f} GB")  # ~64 GB!

# === 4. Speculative Decoding — 2-4× speedup tanpa quality loss ===
# Idea: gunakan model kecil (draft) untuk generate N token, verifikasi dengan model besar
# Draft tokens yang diapprove diterima, yang ditolak di-regenerate dengan model besar
# Speedup karena: model besar bisa verifikasi N draft tokens dalam satu forward pass

def speculative_decode_step(draft_model, target_model, tokenizer, prompt, num_draft=5):
    inputs = tokenizer(prompt, return_tensors="pt")
    draft_output = draft_model.generate(**inputs, max_new_tokens=num_draft, do_sample=False)
    draft_tokens = draft_output[0][inputs["input_ids"].shape[1]:]

    # Verifikasi semua draft tokens dalam satu forward pass model besar
    full_input = torch.cat([inputs["input_ids"], draft_tokens.unsqueeze(0)], dim=-1)
    with torch.no_grad():
        target_logits = target_model(full_input).logits
    return draft_tokens, target_logits

Quantization

INT8: ~2× memory reduction, <1% quality loss. INT4: ~4× reduction, 1–3% loss. NF4 (bitsandbytes) dan GPTQ adalah praktis untuk consumer GPU. AWQ (Activation-aware Weight Quantization) menghasilkan kualitas lebih baik dari GPTQ dengan inference lebih cepat.

KV Cache

Built-in di semua major framework. Memory-intensive untuk context panjang. Optimasi: Grouped Query Attention (GQA, dipakai LLaMA 3) mengurangi jumlah KV heads tanpa much quality loss. Multi-Query Attention (MQA) lebih ekstrem. PagedAttention (vLLM) handles dynamic cache allocation.

Speculative Decoding

2–4× speedup menggunakan draft model kecil + verification model besar. Tidak ada quality loss (mathematically equivalent). Butuh draft model yang distribution-nya mirip target model. Google Medusa dan EAGLE adalah variasi yang lebih efisien.

Continuous Batching

Serve multiple requests in-flight tanpa padding overhead. vLLM dan TensorRT-LLM menggunakan ini untuk production throughput. 10–50× higher throughput vs naive batching. Essential untuk API deployment yang cost-effective.

5.4 Multimodal & Beyond // melampaui teks

LLM modern tidak lagi terbatas pada teks. Kemampuan memahami dan menghasilkan berbagai modalitas — gambar, audio, video, kode, data tabular — telah menjadi frontier aktif dan salah satu differentiator terbesar antar model 2024–2025. Fondasi multimodal bukan hanya menambahkan "input gambar" ke LLM, melainkan membangun shared representasi space di mana konsep yang sama bisa diekspresikan dalam modalitas berbeda.

CLIP (Contrastive Language-Image Pre-training, OpenAI 2021) adalah salah satu contribution terpenting: dengan melatih encoder gambar dan encoder teks secara contrastive pada pasangan gambar-caption dari internet, CLIP menghasilkan embedding space di mana gambar dan deskripsinya berada berdekatan. CLIP menjadi backbone visual encoder untuk hampir semua vision-language model yang mengikutinya.

Arsitektur vision-language model

  • CLIP-style (contrastive) — encoder terpisah untuk gambar dan teks, dilatih dengan contrastive loss. Hasilkan shared embedding space. Sangat baik untuk retrieval dan zero-shot classification. Tidak menghasilkan teks secara langsung.
  • Generative (LLaVA, GPT-4V style) — visual encoder (biasanya CLIP ViT) + projection layer + LLM decoder. Gambar dikonversi menjadi "visual tokens" yang ditambahkan ke konteks LLM. Model bisa menjawab pertanyaan tentang gambar dan menjelaskan konten visual dalam teks.
  • Native multimodal (Gemini, Claude 3 style) — ditraining dari awal dengan data multimodal, bukan gabungan model. Lebih seamless dalam memahami dan menghasilkan konten lintas modalitas.

import torch

for label, prob in zip(labels, probs[0]):

# cat: 0.94, dog: 0.04, car: 0.02 — tanpa training khusus!

5.5 Memilih & Mengevaluasi LLM // panduan praktis 2026

Dengan ratusan LLM tersedia — open source maupun proprietary — memilih model yang tepat untuk use case spesifik bisa terasa overwhelming. Benchmark publik membantu, tapi sering tidak merepresentasikan performa pada task spesifik kamu. Prinsip utama: benchmark publik adalah starting point, bukan endpoint. Selalu evaluasi pada data dan task yang actual kamu gunakan sebelum commit ke production.

Lanskap LLM berubah sangat cepat — model yang state-of-the-art enam bulan lalu sering sudah dilampaui. Di Juni 2026, lini terkuat termasuk Claude Sonnet 4 dan Opus 4 (Anthropic), GPT-4o dan o-series (OpenAI), Gemini 2.5 series (Google), dan beberapa strong open-source models (LLaMA 3.3+, Qwen 3, Mistral Large). Namun patokan yang lebih abadi adalah dimensi evaluasi dan kriteria pemilihan — bukan nama model yang spesifik.

Dimensi evaluasi yang komprehensif

  • Task performance — akurasi pada task spesifik kamu (bukan benchmark umum). Buat eval set dari data production nyata, minimal 100–500 contoh.
  • Latency & throughput — P50, P95, P99 latency. Time-to-first-token (TTFT) untuk streaming. Throughput saat concurrent load. Ini sangat berbeda antar provider bahkan untuk model yang sama.
  • Cost per token — input vs output pricing. Untuk task dengan rasio output/input yang berbeda, model yang "murah" secara headline bisa lebih mahal secara actual workload.
  • Context window — seberapa panjang konteks yang bisa diproses. Penting untuk document processing, code generation, dan long-form analysis.
  • Reliability & consistency — seberapa konsisten output untuk input yang sama? Penting untuk production systems yang butuh determinism.
  • Safety & refusal behavior — seberapa sering model menolak request yang legitimate? Terlalu restrictive bisa jadi masalah sebesar terlalu permissive.

    avg_latency_ms: float
    p95_latency_ms: float
    cost_per_1k: float
    avg_output_tokens: float

async def evaluate_model(model_id: str, test_cases: list[dict]) -> EvalResult:
    """Evaluasi model pada test cases dengan metrik latency, cost, accuracy."""
    client = AsyncOpenAI()  # atau Anthropic() — abstraksi client
    latencies = []
    costs = []
    correct = 0

    for tc in test_cases:
        start = time.perf_counter()
        response = await client.chat.completions.create(
            model=model_id,
            messages=[{"role": "user", "content": tc["input"]}],
            max_tokens=500,
            temperature=0,  # deterministic untuk eval
        )
        latency = (time.perf_counter() - start) * 1000
        latencies.append(latency)

        output = response.choices[0].message.content
        if tc["checker"](output):  # task-specific checker function
            correct += 1

        # Cost calculation (example: GPT-4o pricing)
        input_tokens = response.usage.prompt_tokens
        output_tokens = response.usage.completion_tokens
        cost = (input_tokens * 0.005 + output_tokens * 0.015) / 1000
        costs.append(cost)

    p95 = quantiles(latencies, n=20)[18]  # 95th percentile
    return EvalResult(
        model=model_id,
        task="your_task",
        accuracy=correct / len(test_cases),
        avg_latency_ms=mean(latencies),
        p95_latency_ms=p95,
        cost_per_1k=mean(costs) * 1000,
        avg_output_tokens=mean([tc.usage.completion_tokens for tc in test_cases])
    )

# Run evaluasi paralel untuk beberapa model
async def compare_models():
    models = ["gpt-4o-mini", "gpt-4o", "claude-sonnet-4-6"]
    results = await asyncio.gather(*[
        evaluate_model(m, test_cases) for m in models
    ])
    # Sort by accuracy × (1/cost) untuk efficiency frontier
    return sorted(results, key=lambda r: r.accuracy / r.cost_per_1k, reverse=True)

5.6 Distributed Training // cara melatih model yang tidak muat di satu GPU

LLM modern memiliki puluhan hingga ratusan miliar parameter — satu GPU tidak akan cukup. Distributed training adalah teknik membagi model dan data ke banyak GPU (atau node) agar training tetap feasible. Ada tiga strategi utama yang sering dikombinasikan: Data Parallelism (setiap GPU punya copy model, data dibagi), Tensor Parallelism (satu layer di-split ke multiple GPU), dan Pipeline Parallelism (layer berbeda di GPU berbeda).

Framework PyTorch menyediakan FSDP (Fully Sharded Data Parallel) — sebuah evolusi dari Data Parallel yang tidak hanya membagi data, tapi juga membagi (shard) parameter model, gradient, dan optimizer state ke semua GPU. Ini adalah implementasi dari ZeRO (Zero Redundancy Optimizer) paper dari Microsoft DeepSpeed. ZeRO punya 3 stage: Stage 1 (shard optimizer state), Stage 2 (shard gradients + optimizer), Stage 3 (shard parameters + gradients + optimizer). FSDP ~= ZeRO-3.

Tiga strategi parallelism — kapan pakai yang mana

  • Data Parallelism (DP) — Setiap GPU punya full copy model, data dibagi mini-batch. Gradient di-sync via all-reduce. Simple, tapi boros memory — model harus muat di 1 GPU. Untuk model <1B params dengan GPU besar (A100 80GB), ini sudah cukup.
  • Fully Sharded Data Parallel (FSDP / ZeRO-3) — Parameter, gradient, dan optimizer state di-shard ke semua GPU. Hanya parameter yang dibutuhkan untuk forward/backward pass tertentu yang di-unshard (all-gather) on-the-fly. Bisa training model 10–30× lebih besar dari kapasitas 1 GPU. Overhead komunikasi moderate — sekitar 10–20% overhead vs DP.
  • Tensor Parallelism (TP) — Satu layer Transformer di-split column-wise atau row-wise ke multiple GPU. Berguna untuk layer yang sangat besar (FFN dimension 16K+) yang tidak muat di 1 GPU bahkan setelah FSDP. Overhead komunikasi tinggi (all-reduce setiap forward/backward), jadi biasanya dikombinasikan dengan FSDP (hybrid: FSDP across nodes, TP within node).
  • Pipeline Parallelism (PP) — Layer berbeda di GPU berbeda. GPU 1 handle layer 0-7, GPU 2 handle layer 8-15, dst. Micro-batching mengisi "bubble" idle time. Overhead komunikasi rendah tapi ada pipeline bubbles yang mengurangi efisiensi. Biasanya dikombinasikan dengan TP (3D parallelism: DP + TP + PP).

import torch
import torch.nn as nn
from torch.distributed.fsdp import (
    FullyShardedDataParallel as FSDP,
    MixedPrecision, ShardingStrategy, BackwardPrefetch,
)
from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy
import functools

# === FSDP Configuration untuk Transformer ===
def setup_fsdp(model: nn.Module) -> FSDP:
    """Wrap Transformer model dengan FSDP."""
    # Auto-wrap policy: setiap TransformerBlock jadi FSDP unit terpisah
    transformer_auto_wrap = functools.partial(
        transformer_auto_wrap_policy,
        transformer_layer_cls={TransformerBlock},
    )

    # Mixed precision: forward/backward di bfloat16, reduction di fp32
    mixed_precision = MixedPrecision(
        param_dtype=torch.bfloat16,
        reduce_dtype=torch.float32,
        buffer_dtype=torch.bfloat16,
    )

    model = FSDP(
        model,
        auto_wrap_policy=transformer_auto_wrap,
        sharding_strategy=ShardingStrategy.FULL_SHARD,  # ZeRO-3 equivalent
        mixed_precision=mixed_precision,
        backward_prefetch=BackwardPrefetch.BACKWARD_PRE,  # Overlap comm & compute
        device_id=torch.cuda.current_device(),
    )
    return model

# === Memory breakdown untuk 7B model (tanpa FSDP) ===
def estimate_memory(params_billions: float, seq_len: int, batch_size: int,
                     hidden_dim: int = 4096) -> dict:
    """Estimasi memory usage untuk training LLM."""
    bytes_per_param = 2  # bfloat16
    params_gb = params_billions * 1e9 * bytes_per_param / 1e9
    # Optimizer state: 2× model (Adam: fp32 master weights + fp32 variance)
    optimizer_gb = params_gb * 2 * 2  # fp32 = 4 bytes = 2× bf16
    # Gradient: 1× model (bfloat16)
    gradient_gb = params_gb
    # Activations: roughly hidden_dim * seq_len * batch * num_layers * 4 bytes
    num_layers = max(1, int(params_billions / 0.2))  # rough: 200M per layer
    activation_gb = hidden_dim * seq_len * batch_size * num_layers * 4 / 1e9
    total = params_gb + optimizer_gb + gradient_gb + activation_gb
    return {
        "params_gb": params_gb,
        "optimizer_gb": optimizer_gb,
        "gradient_gb": gradient_gb,
        "activation_gb": activation_gb,
        "total_gb": total,
    }

# === Contoh: LLaMA 7B, seq=2048, batch=4 ===
mem = estimate_memory(7, 2048, 4)
print(f"Memory needed: {mem['total_gb']:.1f} GB")
# Dengan FSDP di 4 GPU: total/4 ~= 25% per GPU
print(f"With FSDP (4 GPUs): ~{mem['total_gb']/4:.1f} GB per GPU")
