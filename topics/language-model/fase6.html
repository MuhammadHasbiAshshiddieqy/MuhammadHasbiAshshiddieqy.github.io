6.1 Prompt Engineering // instruksi yang tepat, output yang berguna

Prompt engineering adalah seni dan sains merancang input ke LLM untuk mendapatkan output yang diinginkan secara konsisten. Meskipun terdengar sederhana, perbedaan antara prompt yang buruk dan baik bisa berarti perbedaan antara aplikasi yang tidak berguna dan produk yang mengesankan. Dengan model modern, prompt sering lebih penting dari fine-tuning — dan jauh lebih murah.

Zero-shot vs Few-shot

Zero-shot: langsung instruksikan task tanpa contoh. Bekerja untuk task sederhana yang sudah banyak ada di training data. "Klasifikasikan sentimen kalimat berikut: ..."

Few-shot (ICL): berikan 2–5 contoh input→output sebelum query sebenarnya. Drastis meningkatkan konsistensi format dan akurasi untuk task non-standar. Rule of thumb: mulai zero-shot, jika inkonsisten tambah contoh.

Chain-of-Thought (CoT)

Minta model "berpikir langkah demi langkah" sebelum menjawab. CoT terbukti meningkatkan akurasi reasoning (matematika, logika, multi-step) secara signifikan. Variannya: Zero-shot CoT — tambahkan "Let's think step by step". Few-shot CoT — berikan contoh dengan reasoning eksplisit. Tree-of-Thought — eksplorasi beberapa jalur reasoning paralel.

Struktur prompt efektif

System: definisikan role dan constraint ("Kamu adalah asisten yang menjawab singkat dan faktual"). Context: berikan latar belakang relevan. Instruction: task yang jelas dan spesifik. Format: tentukan output format (JSON, bullet points, tabel). Constraints: batasi panjang, bahasa, atau scope.

ReAct & Agentic Prompting

ReAct (Reason + Act): model diminta bergantian antara Thought (reasoning), Action (panggil tool), Observation (hasil tool), sampai dapat jawaban final. Ini fondasi dari LLM agents. Prompt ReAct sangat structured: loop eksplisit dengan label Thought/Action/Observation di setiap iterasi.

6.2 RAG — Retrieval-Augmented Generation // knowledge yang selalu up-to-date

LLM punya dua masalah fundamental untuk aplikasi enterprise: knowledge cutoff (tidak tahu event setelah training) dan hallucination (mengarang fakta dengan percaya diri). RAG (Lewis et al., 2020) memecahkan keduanya: alih-alih mengandalkan "memori" model, kita berikan dokumen relevan secara real-time sebagai konteks. Model hanya perlu membaca dan merangkum, bukan mengingat.

Pipeline RAG dasar

  1. Indexing: chunk dokumen → embed → simpan di vector DB
  2. Retrieval: embed query → similarity search → ambil top-K chunks
  3. Augmentation: inject chunks ke prompt sebagai context
  4. Generation: LLM jawab berdasarkan context, bukan memory

Chunking strategy

Fixed-size: potong setiap N token — simple tapi bisa memotong di tengah kalimat. Sentence/paragraph: respek batas semantik — lebih baik untuk retrieval. Recursive: coba split by paragraph → sentence → word secara hierarki. Semantic chunking: embed setiap kalimat, split di mana cosine similarity turun drastis (boundary semantik alami).

Vector Database

Menyimpan embedding dan mendukung approximate nearest neighbor (ANN) search. Pilihan: Chroma (local, gratis, cocok dev), Pinecone (managed, scalable), Weaviate (hybrid search), pgvector (Postgres extension, familiar). ANN algorithm: HNSW (kecepatan tinggi) atau IVF (memory-efficient).

Advanced RAG

Hybrid search: kombinasikan semantic (embedding) + keyword (BM25) — saling melengkapi. Re-ranking: setelah retrieval top-K, gunakan cross-encoder untuk re-rank lebih akurat sebelum masuk LLM. HyDE: generate hypothetical answer dulu, lalu gunakan itu sebagai query embedding. Parent-child chunking: index chunks kecil tapi kirim parent chunk ke LLM.

Pythonrag_pipeline.py
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_chroma import Chroma
from langchain.chains import RetrievalQA

# 1. Indexing
splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=64)
chunks = splitter.create_documents(docs)

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectordb = Chroma.from_documents(chunks, embeddings, persist_directory="./db")

# 2. Retrieval + Generation
retriever = vectordb.as_retriever(search_type="mmr", search_kwargs={"k": 5})
llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

qa_chain = RetrievalQA.from_chain_type(
    llm=llm, retriever=retriever,
    return_source_documents=True
)

result = qa_chain.invoke({"query": "Apa kebijakan refund produk?"})
print(result["result"])
print("Sources:", [d.metadata["source"] for d in result["source_documents"]])

6.3 Function Calling & Structured Output // LLM sebagai orchestrator

Function calling (atau tool use) memungkinkan LLM untuk memanggil fungsi eksternal — database query, API call, calculator, web search — secara terstruktur. Model tidak benar-benar menjalankan fungsi; ia menghasilkan JSON yang berisi nama fungsi dan argumen yang tepat, lalu aplikasi yang menjalankannya dan mengembalikan hasilnya. Ini fondasi dari LLM agents yang bisa berinteraksi dengan dunia nyata.

Alur function calling

  1. Definisikan tool schema (nama, deskripsi, parameter)
  2. Kirim ke LLM bersama user message
  3. LLM putuskan apakah perlu memanggil tool
  4. Jika ya: LLM return tool_call JSON
  5. Aplikasi eksekusi fungsi nyata
  6. Return hasil ke LLM sebagai tool message
  7. LLM generate respons final

Structured Output

Selain tool calling, kamu bisa paksa model output JSON sesuai schema yang ditentukan menggunakan response_format atau Pydantic model. Berguna untuk: ekstraksi data dari teks tak terstruktur, klasifikasi dengan confidence score, parsing dokumen ke database record. OpenAI dan banyak provider mendukung JSON mode atau strict schema enforcement.

Pythonfunction_calling.py
from openai import OpenAI
import json
client = OpenAI()

# Definisikan tools
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Ambil cuaca saat ini untuk kota tertentu",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "Nama kota"},
                "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
            },
            "required": ["city"]
        }
    }
}]

messages = [{"role": "user", "content": "Cuaca Jakarta sekarang gimana?"}]
response = client.chat.completions.create(model="gpt-4o-mini", messages=messages, tools=tools)

# Handle tool call
tool_call = response.choices[0].message.tool_calls[0]
args = json.loads(tool_call.function.arguments)  # {"city": "Jakarta"}

# Eksekusi fungsi nyata
weather_result = get_weather(args["city"])  # {"temp": 32, "condition": "Sunny"}

# Kirim hasil kembali ke LLM
messages += [response.choices[0].message, {"role": "tool", "tool_call_id": tool_call.id, "content": json.dumps(weather_result)}]
final = client.chat.completions.create(model="gpt-4o-mini", messages=messages)
print(final.choices[0].message.content)

# Structured Output dengan Pydantic
from pydantic import BaseModel

class ExtractedOrder(BaseModel):
    product: str; quantity: int; total_price: float; currency: str

result = client.beta.chat.completions.parse(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Saya beli 3 buku seharga Rp150.000"}],
    response_format=ExtractedOrder
)

6.4 LLM Application Development // dari API ke produk

Membangun aplikasi LLM bukan hanya memanggil API. Kamu perlu mengelola conversation state, prompt templates, error handling, retry logic, streaming responses, multi-turn context, dan integrasi dengan berbagai tools. Framework seperti LangChain dan LlamaIndex menyederhanakan ini — tapi penting untuk mengerti apa yang terjadi di balik abstraksi mereka.

LangChain core concepts

Chain: urutan steps (prompt → LLM → output parser). Agent: LLM yang memutuskan sendiri tools apa yang perlu dipanggil dan kapan. Memory: simpan conversation history (buffer, summary, entity). LCEL (LangChain Expression Language): pipe operator | untuk composable chains yang readable dan streamable.

LlamaIndex vs LangChain

LlamaIndex: fokus pada data ingestion dan retrieval — sangat baik untuk RAG, query engine, document Q&A. Abstraksi IndexNode, NodeParser, QueryEngine sangat powerful. LangChain: lebih general purpose — chains, agents, tools, memory semua ada. Lebih verbose tapi lebih fleksibel. Pilih: LlamaIndex untuk data-heavy apps, LangChain untuk agentic workflows.

Streaming & async

User experience jauh lebih baik dengan streaming (karakter muncul satu per satu seperti ChatGPT) dibanding menunggu response penuh. OpenAI SDK mendukung stream=True. Untuk server-side: gunakan Server-Sent Events (SSE) atau WebSocket. FastAPI punya StreamingResponse yang kompatibel.

Conversation management

LLM stateless — setiap request independen. State harus disimpan di aplikasi. Strategi: Full history: kirim semua messages (mahal, ada token limit). Summary memory: summarize lama, detail baru. Vector memory: retrieve hanya conversation relevan. Entity memory: track entitas penting (nama, preferensi user).

Pythonllm_app.py
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate, MessagesPlaceholder
from langchain_core.output_parsers import StrOutputParser
from langchain.memory import ConversationSummaryBufferMemory

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0.7, streaming=True)

prompt = ChatPromptTemplate.from_messages([
    ("system", "Kamu asisten customer service. Jawab ramah dan singkat."),
    MessagesPlaceholder("history"),
    ("human", "{input}")
])
memory = ConversationSummaryBufferMemory(llm=llm, max_token_limit=1000, return_messages=True)

# LCEL chain dengan pipe operator
chain = prompt | llm | StrOutputParser()

async def chat(user_input: str):
    history = memory.load_memory_variables({})["history"]
    # Streaming response
    async for chunk in chain.astream({"input": user_input, "history": history}):
        yield chunk
    memory.save_context({"input": user_input}, {"output": "..."})

# FastAPI endpoint
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
app = FastAPI()

@app.post("/chat")
async def chat_endpoint(req: dict):
    return StreamingResponse(chat(req["message"]), media_type="text/event-stream")
6.5 Guardrails, Safety & Eval Pipeline // LLM yang bisa dipercaya

LLM di production bisa gagal dengan cara yang tidak terduga: hallucination, prompt injection, output berbahaya, atau bias. Tidak cukup mengandalkan sistem alignment dari model provider — kamu perlu lapisan keamanan sendiri. Guardrails adalah checks yang dijalankan sebelum (input) atau sesudah (output) LLM, dan eval pipeline memastikan kualitas tidak turun saat model atau prompt berubah.

Input guardrails

Prompt injection detection: deteksi upaya user untuk override system prompt ("Abaikan instruksi sebelumnya dan..."). PII detection: cegah data sensitif (nomor KTP, nomor kartu kredit) masuk ke LLM. Topic restriction: classifier untuk blokir query di luar scope aplikasi. Rate limiting: cegah abuse dan cost explosion.

Output guardrails

Hallucination detection: verifikasi klaim faktual terhadap sumber (untuk RAG: apakah jawaban ada di retrieved chunks?). Toxicity filter: classifier untuk konten berbahaya sebelum dikirim ke user. Format validation: pastikan output sesuai schema yang diharapkan. Confidence thresholding: jika model tidak yakin, escalate ke human.

LLM Evaluation

Reference-based: bandingkan output dengan ground truth (ROUGE, BLEU untuk summarization). LLM-as-judge: gunakan LLM lain (biasanya GPT-4) untuk menilai kualitas output secara otomatis. RAGAs: framework khusus untuk evaluasi RAG pipeline — faithfulness, answer relevancy, context recall. Human eval: untuk task subjektif, masih yang terbaik tapi mahal.

Continuous eval pipeline

Setiap perubahan prompt atau model harus melewati eval sebelum deploy. Setup: golden dataset (100–500 contoh query + expected output), automated scorer (LLM judge atau metric), regression threshold (blokir deploy jika score turun >5%). Tools: LangSmith, Weights & Biases, PromptFoo.

Pythonguardrails_eval.py
from guardrails import Guard
from guardrails.hub import ToxicLanguage, DetectPII, ValidJson

# Input + Output guardrails
guard = Guard().use_many(
    DetectPII(pii_entities=["CREDIT_CARD", "EMAIL_ADDRESS"], on_fail="exception"),
    ToxicLanguage(threshold=0.5, on_fail="filter"),
    ValidJson(on_fail="reask")  # otomatis re-ask LLM jika output bukan valid JSON
)

validated_output, *rest = guard(openai_call, prompt="...")

# LLM-as-judge evaluator
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_recall

6.6 LLM in Production // latency, cost, observability

Production LLM apps punya tiga musuh: latency (user menunggu terlalu lama), cost (token mahal, terutama di skala), dan reliability (model down, rate limit, output degradasi). Fase ini membahas strategi konkret untuk ketiga masalah itu, plus observability — karena kamu tidak bisa memperbaiki apa yang tidak bisa kamu ukur.

Cost optimization

Model routing: gunakan model kecil (gpt-4o-mini, Haiku) untuk query sederhana, model besar hanya untuk yang kompleks — bisa hemat 80% cost. Prompt caching: provider modern cache prefix prompt — pastikan system prompt di awal dan statis. Output length control: set max_tokens eksplisit. Batch API: untuk offline workload, batch request 50% lebih murah.

Latency optimization

Streaming: time-to-first-token ~200ms, jauh lebih baik dari menunggu full response 5s. Caching semantik: untuk query yang mirip, cache response (GPTCache, Redis). Parallel calls: jika independen, jalankan beberapa LLM call secara async. Smaller models: gpt-4o-mini 5–10× lebih cepat dari gpt-4o. Prompt compression: LLMLingua untuk compress context tanpa kehilangan informasi penting.

Observability

Setiap LLM call harus di-log dengan: input prompt, output, model, latency, token count, cost, user ID, session ID. Tools: LangSmith (terintegrasi dengan LangChain), Langfuse (open source), Helicone (proxy-based). Alert jika: latency p99 > threshold, error rate naik, atau output quality turun.

Reliability patterns

Retry dengan exponential backoff: untuk rate limit errors (429). Fallback chain: primary → secondary provider jika down. Circuit breaker: stop panggil provider yang error rate tinggi. Timeout: set request timeout (30s), jangan biarkan hang. Graceful degradation: tampilkan cached response atau fallback message jika semua gagal.

Pythonllm_production.py
import asyncio, time
from openai import AsyncOpenAI
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type

client = AsyncOpenAI()

# Retry otomatis dengan exponential backoff
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=30),
    retry=retry_if_exception_type(Exception)
)
async def llm_call(messages, model="gpt-4o-mini", timeout=30):
    return await asyncio.wait_for(
        client.chat.completions.create(model=model, messages=messages),
        timeout=timeout
    )

# Model router — hemat cost 80%
def route_model(query: str) -> str:
    if len(query) < 200 and "summarize" not in query.lower():
        return "gpt-4o-mini"   # ~$0.15/1M tokens
    return "gpt-4o"           # ~$2.50/1M tokens
