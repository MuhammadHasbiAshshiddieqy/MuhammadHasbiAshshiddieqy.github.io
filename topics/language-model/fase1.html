1.1 NLP Klasik & Pipeline Teks // sebelum era deep learning

Natural Language Processing (NLP) klasik adalah kumpulan teknik berbasis aturan dan statistik yang dikembangkan jauh sebelum neural network mendominasi. Meski terasa "kuno", pemahaman pipeline klasik krusial — bukan hanya historis, tapi karena banyak konsepnya masih hidup di preprocessing modern, dan karena memahami keterbatasannya adalah cara terbaik untuk mengerti mengapa neural LM lahir.

Pipeline NLP tradisional adalah serangkaian transformasi berurutan: teks mentah diproses melalui tokenisasi, normalisasi, analisis morfologi, tagging sintaktik, hingga pengenalan entitas. Setiap langkah membutuhkan aturan atau model terpisah yang dioptimasi secara manual untuk bahasa tertentu — inilah yang membuatnya tidak scalable lintas bahasa.

Mengapa belajar NLP klasik?LLM modern mengabstraksi banyak langkah ini — tapi pipeline klasik masih dipakai untuk data preprocessing, untuk bahasa dengan resource terbatas, dan sebagai baseline perbandingan. Selain itu, banyak konsep (tokenisasi, stemming, n-gram) muncul kembali dalam wujud modern di dalam LLM itu sendiri.

Komponen pipeline yang perlu dipahami

  • Tokenisasi — memecah teks menjadi unit-unit (token). Trivial untuk Inggris (split by space), kompleks untuk bahasa tanpa spasi (Chinese, Japanese) atau bahasa aglutinatif (Turkish, Finnish). Bahasa Indonesia sendiri punya tantangan karena prefiks/sufiks yang kaya.
  • Stemming vs Lemmatization — stemming memotong akhiran secara mekanik ("running" → "runn"), lemmatization memetakan ke bentuk kamus ("better" → "good"). Lemmatization lebih akurat tapi butuh lexicon.
  • POS Tagging (Part-of-Speech) — memberi label grammatical setiap token: noun, verb, adjective. Dipakai untuk parsing sintaksis dan feature engineering.
  • NER (Named Entity Recognition) — mengidentifikasi entitas bernama: orang, tempat, organisasi, tanggal, nilai uang. Krusial untuk information extraction.
  • Dependency Parsing — menganalisis hubungan gramatikal antar kata dalam kalimat. Lebih dalam dari POS tagging, dipakai untuk question answering dan relation extraction.
Pythonnlp_pipeline.py
import spacy
from nltk.stem import PorterStemmer, WordNetLemmatizer
import nltk

# spaCy: library NLP modern dengan pipeline lengkap
nlp = spacy.load("en_core_web_sm")

text = "Apple's CEO Tim Cook announced new products in San Francisco yesterday."
doc = nlp(text)

# Tokenisasi + POS + Lemmatization
for token in doc:
    print(f"{token.text:15} POS:{token.pos_:8} LEMMA:{token.lemma_:12} STOP:{token.is_stop}")

# NER — Named Entity Recognition
print("\n=== Named Entities ===")
for ent in doc.ents:
    print(f"{ent.text:20} → {ent.label_} ({spacy.explain(ent.label_)})")
# Output: Apple → ORG, Tim Cook → PERSON, San Francisco → GPE

# Stemming manual dengan NLTK (lebih agresif dari lemmatization)
stemmer = PorterStemmer()
words = ["running", "better", "organizational", "generalization"]
for w in words:
    print(f"{w:20} → stem: {stemmer.stem(w)}")
# running → run, better → better, organizational → organiz

# Dependency parsing — siapa subject, verb, object?
print("\n=== Dependency Tree ===")
for token in doc:
    if token.dep_ in ["nsubj", "dobj", "ROOT"]:
        print(f"{token.dep_:10} → '{token.text}' (head: '{token.head.text}')")

Perbandingan: NLP Klasik vs Modern

  • Klasik — pipeline modular, rule-based/statistical, butuh feature engineering manual, tidak transfer well lintas domain, deterministic dan interpretable
  • Modern (neural) — end-to-end learned, transfer learning, butuh data besar, black box, tapi jauh lebih akurat dan generalizable
  • Hybrid — kombinasi keduanya masih umum: preprocessing dengan spaCy, lalu feed ke BERT/GPT. Tokenisasi modern (BPE) pun mewarisi intuisi dari tokenisasi klasik
  • Kapan pakai klasik — dataset kecil, butuh determinism (auditable), resource terbatas (embedded system), bahasa dengan resource training minim

Latihan

1.2 Representasi Teks // dari kata ke angka

Machine learning hanya bisa bekerja dengan angka — tapi teks adalah simbol. Representasi teks adalah jembatan antara dunia linguistik dan dunia numerik. Cara kita merepresentasikan teks secara fundamental menentukan apa yang bisa dipelajari model dari data tersebut.

Evolusi representasi teks mencerminkan evolusi pemahaman kita tentang bahasa: dari one-hot encoding yang sepenuhnya mengabaikan makna, hingga TF-IDF yang memperhitungkan pentingnya kata dalam corpus, hingga dense embeddings yang menangkap hubungan semantik secara implisit.

One-Hot Encoding

Setiap kata direpresentasikan sebagai vektor biner dengan dimensi sebesar ukuran vocabulary. Hanya satu posisi yang bernilai 1. Masalah: vektor sangat sparse, tidak ada informasi semantik, "cat" dan "dog" sama jauhnya dengan "cat" dan "car".

Bag of Words (BoW)

Dokumen direpresentasikan sebagai vektor frekuensi kata, mengabaikan urutan. "The cat sat" dan "The sat cat" identik. Berguna untuk klasifikasi teks sederhana tapi kehilangan konteks posisi dan semantik.

TF-IDF

Term Frequency × Inverse Document Frequency. Kata yang sering muncul di satu dokumen tapi jarang di corpus lain diberi bobot tinggi. Lebih informatif dari BoW — kata seperti "the" yang ada di mana-mana mendapat bobot rendah otomatis.

Sparse vs Dense

BoW dan TF-IDF menghasilkan vektor sparse (ribuan dimensi, kebanyakan nol). Word embeddings menghasilkan vektor dense (100–300 dimensi, semua terisi). Dense lebih efisien komputasi dan menangkap similaritas semantik.

Keterbatasan representasi berbasis frekuensiTF-IDF tidak menangkap polisemi (satu kata, banyak makna: "bank" sebagai institusi vs tepi sungai) dan tidak menangkap sinonim ("car" dan "automobile" terlihat sangat berbeda). Keterbatasan ini mendorong lahirnya word embeddings.
Pythontext_representation.py
from sklearn.feature_extraction.text import CountVectorizer, TfidfVectorizer
import numpy as np

corpus = [
    "the cat sat on the mat",
    "the dog sat on the rug",
    "cats and dogs are animals",
]

# === Bag of Words ===
bow = CountVectorizer()
bow_matrix = bow.fit_transform(corpus)
print("Vocabulary:", bow.get_feature_names_out())
print("BoW matrix shape:", bow_matrix.shape)  # (3, 10) — sparse
print("Doc 1:", bow_matrix[0].toarray())

# === TF-IDF ===
tfidf = TfidfVectorizer()
tfidf_matrix = tfidf.fit_transform(corpus)

# Kata "the" muncul di semua dokumen → IDF rendah → bobot rendah
# Kata "cat" hanya di doc 1 → IDF tinggi → bobot tinggi
feature_names = tfidf.get_feature_names_out()
doc1_tfidf = tfidf_matrix[0].toarray()[0]

# Sort berdasarkan bobot — kata paling "penting" di doc 1
top_words = sorted(zip(feature_names, doc1_tfidf), key=lambda x: x[1], reverse=True)[:5]
print("Top TF-IDF words in doc 1:", top_words)
# [('cat', 0.49), ('mat', 0.49), ('sat', 0.37), ('on', 0.37), ('the', 0.25)]

# === Cosine Similarity antara dokumen ===
from sklearn.metrics.pairwise import cosine_similarity
sim = cosine_similarity(tfidf_matrix)
print(f"\nSimilarity doc1 vs doc2: {sim[0,1]:.3f}")  # tinggi — sama-sama "sat on the"
print(f"Similarity doc1 vs doc3: {sim[0,2]:.3f}")  # rendah — vocabulary berbeda

1.3 Word Embeddings // geometri makna

Word embeddings adalah salah satu ide paling elegan dalam NLP: representasikan setiap kata sebagai titik dalam ruang vektor berdimensi rendah (biasanya 100–300 dimensi), sedemikian hingga kata-kata dengan makna serupa berada berdekatan satu sama lain. Hasilnya adalah geometri makna yang bisa dikuantifikasi — dan bahkan diaritmetikakan.

Kunci intuisi di balik word embeddings adalah distributional hypothesis: kata-kata yang muncul dalam konteks yang sama cenderung memiliki makna yang sama ("a word is known by the company it keeps" — Firth, 1957). Word2Vec mengeksploitasi hipotesis ini secara langsung: dengan melatih neural network sederhana untuk memprediksi kata berdasarkan konteksnya, embedding yang dihasilkan secara otomatis menangkap hubungan semantik dan sintaktik.

Word arithmetic yang terkenalking - man + woman ≈ queen. Ini bukan magic — ini konsekuensi dari struktur ruang vektor yang dipelajari. Dimensi-dimensi tertentu dalam ruang embedding menangkap konsep seperti "royalty", "gender", "animacy". Fenomena ini menjadi bukti pertama bahwa representasi neural bisa menangkap struktur semantik secara implisit.

Dua arsitektur Word2Vec

  • CBOW (Continuous Bag of Words) — prediksi kata target dari window konteks di sekitarnya. Rata-rata vektor konteks, prediksi kata tengah. Lebih cepat dilatih, bekerja baik untuk kata yang sering muncul.
  • Skip-gram — kebalikannya: dari kata target, prediksi kata-kata di window konteksnya. Lebih lambat dilatih tapi representasi lebih baik untuk kata jarang (rare words). Default yang disarankan untuk dataset besar.
  • GloVe (Global Vectors) — pendekatan berbeda: factorize matriks co-occurrence global. Menggabungkan kekuatan statistik global (seperti LSA) dengan pembelajaran lokal konteks (seperti Word2Vec). Umumnya sedikit lebih baik dari Word2Vec pada tugas semantic similarity.
  • FastText — ekstensi Word2Vec dari Facebook yang memecah kata menjadi character n-grams. Keunggulan utama: bisa menghasilkan embedding untuk kata yang tidak ada di training vocabulary (OOV), dan representasinya lebih baik untuk bahasa morfologis kaya seperti Bahasa Indonesia atau Turki.
Pythonword_embeddings.py
from gensim.models import Word2Vec, KeyedVectors
import numpy as np

# Training Word2Vec dari scratch
sentences = [
    ["the", "cat", "sat", "on", "the", "mat"],
    ["the", "dog", "ran", "in", "the", "park"],
    # ... ribuan kalimat dari corpus
]

model = Word2Vec(
    sentences,
    vector_size=100,    # dimensi embedding
    window=5,           # context window size
    min_count=5,        # abaikan kata yang muncul < 5x
    workers=4,          # parallelism
    sg=1,               # 1=skip-gram, 0=CBOW
    epochs=10,
)

# Load pre-trained Word2Vec (Google News, 3M words, 300 dim)
# wv = KeyedVectors.load_word2vec_format('GoogleNews-vectors-negative300.bin', binary=True)

wv = model.wv  # KeyedVectors — hanya weights, tanpa training overhead

# Semantic similarity
print(wv.similarity("cat", "dog"))    # tinggi — keduanya hewan peliharaan
print(wv.similarity("cat", "car"))    # rendah

# Most similar words
print(wv.most_similar("king", topn=5))

# Word arithmetic: king - man + woman = ?
result = wv.most_similar(
    positive=["king", "woman"],
    negative=["man"],
    topn=3
)
print("king - man + woman =", result)  # → [('queen', 0.71), ...]

# Mendeteksi kata yang "tidak sekelompok"
print(wv.doesnt_match(["cat", "dog", "fish", "laptop"]))  # → laptop

# Visualisasi dengan PCA (dimensionality reduction ke 2D)
from sklearn.decomposition import PCA
words = ["king", "queen", "man", "woman", "cat", "dog"]
vectors = np.array([wv[w] for w in words])

1.4 Language Modeling Klasik // prediksi kata dengan statistik

Language model adalah model probabilistik yang mengassign probabilitas pada suatu urutan kata. Pertanyaan yang dijawab: "Seberapa mungkin kalimat ini ada dalam bahasa X?" atau lebih pratisnya: "Kata apa yang paling mungkin muncul setelah urutan kata ini?" Ini adalah tugas yang sama yang dilakukan LLM modern — perbedaannya hanya pada cara menjawabnya.

N-gram language model adalah pendekatan klasik yang paling luas digunakan sebelum era neural. Prinsipnya sederhana: probabilitas kata berikutnya hanya bergantung pada N-1 kata sebelumnya (Markov assumption). Meski terlihat naif, N-gram model menjadi backbone sistem speech recognition, machine translation, dan spell checking selama dekade.

Pythonngram_lm.py
from collections import defaultdict, Counter
import math
import random

class NgramLM:
    """Bigram Language Model dengan Laplace smoothing."""
    def __init__(self, n: int = 2):
        self.n = n
        self.ngrams = defaultdict(Counter)  # context → {next_word: count}
        self.vocab = set()

    def train(self, corpus: list[list[str]]):
        for sentence in corpus:
            tokens = ["<s>"] * (self.n - 1) + sentence + ["</s>"]
            self.vocab.update(sentence)
            for i in range(len(tokens) - self.n + 1):
                context = tuple(tokens[i:i+self.n-1])
                next_word = tokens[i+self.n-1]
                self.ngrams[context][next_word] += 1

    def prob(self, word: str, context: tuple) -> float:
        # Laplace (add-1) smoothing: P(w|ctx) = (count+1)/(total+|V|)
        counts = self.ngrams[context]
        total = sum(counts.values())
        V = len(self.vocab)
        return (counts[word] + 1) / (total + V)

    def perplexity(self, test_corpus: list[list[str]]) -> float:
        """Perplexity: ukuran seberapa "terkejut" model melihat test data.
        Perplexity rendah = model bagus. PP = 2^(average negative log-prob)."""
        log_prob_sum = 0
        N = 0
        for sentence in test_corpus:
            tokens = ["<s>"] * (self.n - 1) + sentence + ["</s>"]
            for i in range(self.n - 1, len(tokens)):
                ctx = tuple(tokens[i-self.n+1:i])
                word = tokens[i]
                log_prob_sum += math.log2(self.prob(word, ctx))
                N += 1
        return 2 ** (-log_prob_sum / N)

    def generate(self, max_len: int = 20) -> str:
        tokens = ["<s>"] * (self.n - 1)
        while len(tokens) < max_len:
            ctx = tuple(tokens[-(self.n-1):])
            candidates = list(self.ngrams[ctx].keys())
            weights = list(self.ngrams[ctx].values())
            if not candidates: break
            next_word = random.choices(candidates, weights=weights)[0]
            if next_word == "</s>": break
            tokens.append(next_word)
        return " ".join(tokens[self.n-1:])

Masalah fundamentall N-gram model

  • Sparsity problem — sebagian besar N-gram yang valid secara linguistik tidak pernah muncul di training data. Smoothing (Laplace, Kneser-Ney, Good-Turing) merupakan cara mengatasi ini, tapi tidak sempurna.
  • Long-range dependency — kalimat "The man who works at the factory where my brother used to work is tired." — model bigram tidak bisa menangkap bahwa "is" harus agree dengan "man", bukan "work".
  • Curse of dimensionality — untuk vocabulary V, trigram model butuh V³ parameter. Dengan V=50.000, ini 125 miliar parameter hanya untuk trigram — tidak tractable.
  • Tidak ada generalisasi — "dog eats food" dan "cat eats food" adalah trigram berbeda sepenuhnya, meski maknanya hampir sama. Model tidak menyadari "dog" dan "cat" mirip.
Perplexity sebagai metrik intrinsikPerplexity mengukur rata-rata jumlah pilihan yang dipertimbangkan model per token. PP=100 berarti "model seolah memilih dari 100 kata secara acak setiap langkah". Model bahasa inggris yang bagus punya PP 50–100 pada test set standard. Perplexity berguna untuk membandingkan model dalam setting yang sama, tapi tidak selalu berkorelasi dengan performa downstream task.

1.5 Subword Tokenization // solusi OOV yang elegan

Semua model bahasa modern — GPT, BERT, T5 — menggunakan subword tokenization. Ini bukan detail implementasi kecil; ini adalah keputusan arsitektur fundamental yang menentukan bagaimana model "melihat" bahasa. Memahami subword tokenization adalah kunci untuk memahami mengapa model berperilaku seperti yang kita lihat.

Problem yang diselesaikan: word-level tokenization menghasilkan vocabulary terlalu besar (jutaan kata termasuk variasi morfologi dan misspelling) dengan banyak OOV (out-of-vocabulary). Character-level tokenization tidak mengalami OOV tapi sequence jadi sangat panjang dan sulit dipelajari. Subword tokenization adalah jalan tengah yang elegan: pecah kata menjadi sub-unit yang sering muncul, sehingga semua kata bisa direpresentasikan dari vocabulary yang manageable.

Tiga algoritma utama

  • BPE (Byte-Pair Encoding) — mulai dari karakter individual, iteratif merge pasangan karakter yang paling sering muncul bersamaan. "low", "lower", "newest" → ["low", "er", "new", "est"]. Digunakan oleh GPT series.
  • WordPiece — mirip BPE tapi merge berdasarkan likelihood (bukan frekuensi). Prefix "##" menandai sub-token yang bukan awal kata. "playing" → ["play", "##ing"]. Digunakan oleh BERT dan Google.
  • SentencePiece / Unigram LM — model berbasis probabilistik yang dimulai dari vocabulary besar dan memangkas. Tidak butuh pre-tokenization berbasis spasi — language-agnostic. Digunakan oleh T5, LLaMA, Gemini.
Pythontokenization.py
from tokenizers import Tokenizer
from tokenizers.models import BPE
from tokenizers.trainers import BpeTrainer
from tokenizers.pre_tokenizers import Whitespace
from transformers import AutoTokenizer

# === Training BPE Tokenizer dari Scratch ===
tokenizer = Tokenizer(BPE(unk_token="[UNK]"))
tokenizer.pre_tokenizer = Whitespace()

trainer = BpeTrainer(
    vocab_size=5000,
    special_tokens=["[UNK]", "[CLS]", "[SEP]", "[PAD]"],
    min_frequency=2
)

files = ["corpus.txt"]
tokenizer.train(files, trainer)
tokenizer.save("bpe_tokenizer.json")

# === Memakai tokenizer GPT-2 (BPE) ===
gpt2_tok = AutoTokenizer.from_pretrained("gpt2")
tokens = gpt2_tok.tokenize("tokenization is fascinating!")
print(tokens)
# ['token', 'ization', ' is', ' fascinating', '!']

ids = gpt2_tok.encode("tokenization is fascinating!")
print(ids)  # [30001, 1634, 318, 13413, 0]
print(gpt2_tok.decode(ids))  # reconstruct original text

# === Memakai tokenizer BERT (WordPiece) ===
bert_tok = AutoTokenizer.from_pretrained("bert-base-uncased")
tokens = bert_tok.tokenize("unaffordable generalization")
print(tokens)
# ['un', '##afford', '##able', 'general', '##ization']
# Note ##prefix → sub-token bukan di awal kata

# === Vocabulary size trade-off ===
for size in [1000, 8000, 32000, 100000]:
    print(f"vocab={size:6d}: avg tokens per word ≈ {estimate_fertility(size):.2f}")
# Fertility rendah = efisien. GPT-4 menggunakan ~100K vocab untuk fertility ~1.3 pada English

↑ Fertility = rata-rata jumlah token per kata. Vocab besar → lebih efisien tapi embedding matrix lebih besar. Sweet spot biasanya 32K–100K untuk multilingual.

Tokenizer mismatch adalah bug yang halusSelalu gunakan tokenizer yang sama dengan model yang kamu pakai. Menghitung token "manual" dengan len(text.split()) bisa off 2–5x dari jumlah token sebenarnya, terutama untuk kode, angka, atau bahasa non-Inggris. Gunakan tokenizer.encode(text) untuk count yang akurat sebelum mengestimasi cost API.

1.6 Evaluasi NLP // mengukur apa yang susah diukur

Evaluasi adalah salah satu masalah tersulit dalam NLP — lebih sulit dari membangun modelnya sendiri. Teks adalah output open-ended: untuk satu input bisa ada ratusan output yang "benar" dan baik, dan kualitas seringkali bersifat subjektif. Metrik otomatis adalah proxy yang tidak sempurna, human evaluation mahal dan tidak scalable, dan benchmark sering di-overfit.

Pemahaman metrik evaluasi bukan sekadar akademik — ini langsung mempengaruhi keputusan engineering: kapan model cukup baik untuk production, bagaimana membandingkan dua model, dan bagaimana mendeteksi regresi saat finetuning. Salah memilih metrik bisa membuat kamu optimize hal yang salah.

Metrik yang wajib diketahui

  • BLEU (Bilingual Evaluation Understudy) — mengukur overlap n-gram antara teks yang dihasilkan dan referensi. Standard untuk machine translation. Masalah: tidak sensitif terhadap parafrase, menghukum variasi linguistik yang sah. Score 0.3 bisa sangat baik atau sangat buruk tergantung task.
  • ROUGE (Recall-Oriented Understudy for Gisting Evaluation) — mirip BLEU tapi fokus pada recall. ROUGE-N mengukur n-gram overlap, ROUGE-L mengukur longest common subsequence. Standard untuk summarization.
  • Perplexity — metrik intrinsik untuk language modeling. Sudah dibahas di topik sebelumnya. Berguna untuk membandingkan model tapi tidak langsung berkorelasi dengan downstream task quality.
  • BERTScore — menggunakan contextual embeddings dari BERT untuk mengukur similarity semantik. Lebih korelasi dengan human judgment dibanding BLEU/ROUGE karena memahami parafrase. Lebih lambat dihitung.
  • Human Evaluation — gold standard. Mahal, tidak scalable, rentan inter-annotator disagreement. Biasanya dilakukan pada sampel kecil sebagai kalibrasi untuk metrik otomatis.
Pythonevaluation_metrics.py
from nltk.translate.bleu_score import sentence_bleu, SmoothingFunction
from rouge_score import rouge_scorer
from bert_score import score as bert_score

# === BLEU Score ===
reference = [["the", "cat", "is", "on", "the", "mat"]]
hypothesis1 = ["the", "cat", "is", "on", "the", "mat"]   # perfect
hypothesis2 = ["the", "feline", "rests", "on", "the", "rug"]  # paraphrase

smooth = SmoothingFunction().method1
print(f"BLEU (perfect): {sentence_bleu(reference, hypothesis1, smoothing_function=smooth):.3f}")
print(f"BLEU (paraphrase): {sentence_bleu(reference, hypothesis2, smoothing_function=smooth):.3f}")
# paraphrase = ~0.13 meski maknanya hampir sama — ini masalah BLEU

# === ROUGE Score ===
scorer = rouge_scorer.RougeScorer(['rouge1', 'rouge2', 'rougeL'], use_stemmer=True)

reference_text = "The president signed the bill into law yesterday."
generated_summary = "Yesterday, the bill was signed by the president."

scores = scorer.score(reference_text, generated_summary)
for metric, result in scores.items():
    print(f"{metric}: P={result.precision:.3f} R={result.recall:.3f} F1={result.fmeasure:.3f}")

# === BERTScore ===
candidates = ["The feline rests on the floor mat."]
references  = ["The cat is lying on the mat."]
P, R, F1 = bert_score(candidates, references, lang="en", verbose=False)
print(f"BERTScore F1: {F1.mean():.3f}")  # ~0.90 — menangkap semantic similarity
# vs BLEU yang hanya ~0.15 untuk kalimat yang sama
Benchmark overfitting adalah masalah nyataBanyak model modern secara tidak sengaja "belajar" benchmark tertentu karena data evaluasi bocor ke training set, atau karena model dioptimasi terlalu spesifik untuk benchmark tersebut. Skor MMLU tinggi tidak selalu berarti model lebih berguna di dunia nyata. Selalu uji dengan data internal dan use case spesifik kamu.
