2.1 Serving Fundamentals // online, batch, streaming

Sebelum menulis satu baris kode serving, kamu harus memahami satu pertanyaan mendasar: bagaimana model ini akan digunakan? Jawabannya menentukan hampir seluruh arsitektur serving — dari infrastruktur hingga optimasi yang tepat. Ada tiga paradigma utama: online inference, batch inference, dan streaming inference — dan memilih yang salah bisa berdampak besar pada cost dan user experience.

Online inference (real-time) adalah model yang dipanggil langsung sebagai respons terhadap request pengguna — misalnya autocomplete, fraud detection, atau recommendation. Constraint utamanya adalah latensi: pengguna tidak mau menunggu lebih dari 100–300ms. Batch inference adalah kebalikannya — kamu memproses ribuan hingga jutaan data sekaligus pada jadwal tertentu (nightly pipeline, weekly scoring). Constraint utamanya adalah throughput dan cost: latency tidak kritis, tapi kamu ingin memproses sebanyak mungkin data per dollar. Streaming inference adalah hybrid — data mengalir dari event stream (Kafka, Kinesis) dan diproses near-real-time, tanpa pengguna menunggu secara langsung tapi tetap butuh SLA tertentu.

Memahami SLA (Service Level Agreement) adalah fondasi semua keputusan teknis. SLA bukan hanya target latency rata-rata — yang relevan untuk user experience adalah percentile tinggi: P95 dan P99 latency. Jika P50 latency kamu 50ms tapi P99 adalah 2 detik, 1% pengguna mendapat pengalaman buruk — dan dalam sistem bertraffic tinggi, 1% bisa berarti jutaan request per hari.

SLA: lebih dari sekadar angka rata-rata

  • P50 (median) — 50% request lebih cepat dari ini. Angka yang "bagus" di dashboard tapi menyembunyikan tail latency.
  • P95 — 95% request selesai dalam waktu ini. Target umum untuk user-facing API: <200ms.
  • P99 — 99% request. Yang sering memicu PagerDuty. Dalam sistem bertraffic tinggi, P99 = pengalaman ribuan pengguna per menit.
  • P999 (P99.9) — relevan untuk high-stakes systems (payment, medical). 1 dari 1000 request — jangan abaikan ini di sistem kritis.

Formula Little's Law memberikan insight penting: L = λ × W — jumlah request in-flight (L) sama dengan arrival rate (λ) dikali rata-rata waktu di sistem (W). Jika throughput kamu 100 QPS dan latency 200ms, rata-rata ada 20 request sedang diproses secara bersamaan. Ini menentukan berapa banyak worker/replica yang kamu butuhkan.

Pythonserving_modes.py
import numpy as np
from dataclasses import dataclass
from typing import List
import time

# === Online Inference: single-request, latency-optimized ===
class OnlinePredictor:
    """Latency-first: proses satu request secepat mungkin."""
    def __init__(self, model, device="cuda"):
        self.model = model.to(device).eval()
        self.device = device

    def predict(self, features: np.ndarray) -> dict:
        import torch
        with torch.no_grad():
            x = torch.tensor(features, dtype=torch.float32).unsqueeze(0).to(self.device)
            # torch.cuda.synchronize() untuk measurement akurat
            start = time.perf_counter()
            output = self.model(x)
            latency_ms = (time.perf_counter() - start) * 1000
            return {"prediction": output.cpu().numpy(), "latency_ms": latency_ms}

# === Batch Inference: throughput-first ===
class BatchPredictor:
    """Throughput-first: proses banyak data sekaligus, gunakan GPU penuh."""
    def __init__(self, model, batch_size=256, device="cuda"):
        self.model = model.to(device).eval()
        self.batch_size = batch_size
        self.device = device

    def predict_dataset(self, dataset) -> List[np.ndarray]:
        import torch
        from torch.utils.data import DataLoader
        loader = DataLoader(dataset, batch_size=self.batch_size, num_workers=4, pin_memory=True)
        results = []
        with torch.no_grad():
            for batch in loader:
                x = batch.to(self.device, non_blocking=True)
                out = self.model(x)
                results.append(out.cpu().numpy())
        return np.concatenate(results)

# === SLA Monitoring: track percentile latencies ===
class LatencyTracker:
    def __init__(self, window_size: int = 1000):
        self.latencies = []
        self.window_size = window_size

    def record(self, latency_ms: float):
        self.latencies.append(latency_ms)
        if len(self.latencies) > self.window_size:
            self.latencies.pop(0)

    def report(self) -> dict:
        arr = np.array(self.latencies)
        return {
            "p50":  np.percentile(arr, 50),
            "p95":  np.percentile(arr, 95),
            "p99":  np.percentile(arr, 99),
            "mean": arr.mean(),
            "max":  arr.max(),
        }
Jangan ukur latency dengan time.time()Untuk GPU, time.perf_counter() sebelum dan sesudah forward pass tidak akurat karena CUDA bersifat asynchronous — kernel mungkin belum selesai saat kamu mengukur. Selalu panggil torch.cuda.synchronize() sebelum stop timer, atau gunakan torch.cuda.Event untuk pengukuran yang benar-benar akurat. Di production, gunakan server-side latency metric yang diukur end-to-end dari request masuk hingga response keluar.
Pythongpu_latency_measurement.py
import torch

def measure_gpu_latency(model, input_tensor, n_warmup=10, n_runs=100) -> dict:
    """Pengukuran latency GPU yang benar menggunakan CUDA Events."""
    model.eval()
    input_tensor = input_tensor.cuda()

    # Warmup: GPU perlu beberapa run pertama untuk "warm up" CUDA kernels
    with torch.no_grad():
        for _ in range(n_warmup):
            _ = model(input_tensor)
    torch.cuda.synchronize()

    # Actual measurement dengan CUDA Events
    latencies = []
    with torch.no_grad():
        for _ in range(n_runs):
            start_event = torch.cuda.Event(enable_timing=True)
            end_event   = torch.cuda.Event(enable_timing=True)
            start_event.record()
            _ = model(input_tensor)
            end_event.record()
            torch.cuda.synchronize()
            latencies.append(start_event.elapsed_time(end_event))

    import numpy as np
    arr = np.array(latencies)
    return {
        "p50_ms": float(np.percentile(arr, 50)),
        "p95_ms": float(np.percentile(arr, 95)),
        "p99_ms": float(np.percentile(arr, 99)),
        "mean_ms": float(arr.mean()),
    }

2.2 FastAPI untuk ML // REST API yang production-ready

FastAPI bukan sekadar framework web — untuk ML serving, ia adalah titik pertemuan antara dunia Python/ML dan dunia HTTP/produksi. Yang membedakan FastAPI dari Flask atau Django untuk use case ini adalah tiga hal: async-first (bisa handle ratusan concurrent request tanpa blocking), Pydantic integration (validasi dan dokumentasi schema otomatis), dan type hints yang membuat API contract eksplisit dan mudah di-test.

Tapi perhatian utama bukan tentang feature — ini tentang arsitektur. Model ML adalah resource yang mahal: loading model membutuhkan waktu (bisa 5–30 detik untuk model besar), dan model harus di-share antar request, bukan dibuat ulang untuk setiap request. Pola yang benar adalah load model sekali saat startup, shared sebagai global state atau dependency injection. Ini kedengarannya sederhana tapi sangat sering salah diimplementasikan.

Pertimbangan lain yang krusial: FastAPI + Uvicorn secara default adalah single-process. Untuk production dengan CPU-bound ML inference, kamu butuh Gunicorn sebagai process manager dengan multiple Uvicorn workers — atau lebih baik lagi, containerize dan horizontal scale di Kubernetes.

Pythonml_api.py
from contextlib import asynccontextmanager
from typing import List, Optional
import numpy as np
import torch

from fastapi import FastAPI, HTTPException, Depends
from pydantic import BaseModel, Field, field_validator

# === Schema dengan Pydantic v2 ===
class PredictRequest(BaseModel):
    features: List[List[float]] = Field(
        ...,
        description="2D list of feature vectors",
        min_length=1,
        max_length=256,
    )
    model_version: Optional[str] = Field(None, description="Specific model version")

    @field_validator("features")
    @classmethod
    def validate_features(cls, v):
        for row in v:
            if len(row) != 128:  # expected feature dim
                raise ValueError(f"Feature vector harus 128-dim, dapat {len(row)}")
        return v

class PredictResponse(BaseModel):
    predictions: List[int]
    probabilities: List[List[float]]
    latency_ms: float
    model_version: str

# === Model state — dibuat sekali, dipakai semua request ===
class ModelState:
    model: torch.nn.Module = None
    device: str = "cpu"
    version: str = "unknown"

model_state = ModelState()

# === Lifespan: load model saat startup, cleanup saat shutdown ===
@asynccontextmanager
async def lifespan(app: FastAPI):
    # STARTUP: load model ke memory/GPU sekali saja
    print("Loading model...")
    model_state.device = "cuda" if torch.cuda.is_available() else "cpu"
    model_state.model = load_model("models/classifier_v2.pt")
    model_state.model = model_state.model.to(model_state.device).eval()
    model_state.version = "v2.1.0"
    print(f"Model ready on {model_state.device}")
    yield
    # SHUTDOWN: cleanup resources
    del model_state.model
    torch.cuda.empty_cache()

app = FastAPI(title="ML Prediction API", version="2.1.0", lifespan=lifespan)

# === Prediction endpoint ===
@app.post("/predict", response_model=PredictResponse)
async def predict(req: PredictRequest):
    if model_state.model is None:
        raise HTTPException(status_code=503, detail="Model not loaded")
    import time
    x = torch.tensor(req.features, dtype=torch.float32).to(model_state.device)
    t0 = time.perf_counter()
    with torch.no_grad():
        logits = model_state.model(x)
        probs = torch.nn.functional.softmax(logits, dim=-1)
    latency_ms = (time.perf_counter() - t0) * 1000
    return PredictResponse(
        predictions=probs.argmax(dim=-1).tolist(),
        probabilities=probs.tolist(),
        latency_ms=latency_ms,
        model_version=model_state.version,
    )

@app.get("/health")
async def health():
    return {"status": "ok", "model_loaded": model_state.model is not None}
Async vs Sync endpoint untuk ML inferenceUntuk CPU-bound ML inference, async def endpoint di FastAPI tidak memberikan concurrency nyata — Python GIL tetap membatasi. Gunakan asyncio.get_event_loop().run_in_executor(None, sync_fn) atau run inference di thread pool untuk tidak memblokir event loop. Untuk GPU inference yang lambat (>50ms), selalu offload ke thread pool agar FastAPI bisa terus melayani health check dan request lain saat inference berjalan.

import asyncio
from concurrent.futures import ThreadPoolExecutor
from fastapi import FastAPI
import torch

# Thread pool untuk CPU-bound inference
executor = ThreadPoolExecutor(max_workers=4)

def run_inference_sync(model, x: torch.Tensor) -> torch.Tensor:
    """Sync function yang dijalankan di thread pool."""
    with torch.no_grad():
        return model(x)

@app.post("/predict/async")
async def predict_async(req: PredictRequest):
    """Async endpoint yang tidak memblokir event loop saat inference."""
    x = torch.tensor(req.features, dtype=torch.float32)
    loop = asyncio.get_event_loop()
    # run_in_executor: jalankan di thread pool, await hasilnya
    output = await loop.run_in_executor(
        executor,
        run_inference_sync,
        model_state.model,
        x
    )
    return {"predictions": output.argmax(dim=-1).tolist()}

2.3 Triton Inference Server // serving skala enterprise

NVIDIA Triton Inference Server adalah solusi serving yang dirancang untuk satu tujuan spesifik: memaksimalkan GPU utilization untuk model inference di skala enterprise. Ketika FastAPI dengan satu model mungkin menggunakan GPU 20–30% (karena GPU menunggu request), Triton bisa mendorong utilisasi hingga 80–95% melalui beberapa mekanisme kunci yang tidak ada di serving framework generik.

Yang pertama adalah dynamic batching: Triton secara otomatis mengumpulkan request yang datang hampir bersamaan dan memproses mereka dalam satu batch GPU call. Request dari client A yang datang 2ms lebih awal bisa digabung dengan request dari client B — keduanya dapat response dalam waktu hampir sama, tapi GPU digunakan jauh lebih efisien. Yang kedua adalah concurrent model execution: Triton bisa menjalankan multiple instance model sekaligus pada GPU yang sama, memanfaatkan paralelisme hardware yang tidak bisa dilakukan oleh single-process server.

Triton mendukung semua backend populer secara native: PyTorch TorchScript, ONNX Runtime, TensorRT, TensorFlow SavedModel, Python (untuk preprocessing/postprocessing), dan bahkan custom C++ backend. Model ensemble memungkinkan kamu merangkai pre/post-processing dan multiple models dalam satu pipeline yang dipanggil sebagai satu request dari client.

Pythontriton_client.py
import numpy as np
import tritonclient.http as httpclient
from tritonclient.utils import InferenceServerException

class TritonMLClient:
    def __init__(self, url: str = "localhost:8000", model_name: str = "text_classifier"):
        self.client = httpclient.InferenceServerClient(url=url)
        self.model_name = model_name
        # Verifikasi model tersedia dan ready
        assert self.client.is_model_ready(model_name), f"Model {model_name} not ready"

    def predict(self, input_ids: np.ndarray) -> np.ndarray:
        """Send inference request ke Triton."""
        # Buat input tensor
        inputs = [httpclient.InferInput("input_ids", input_ids.shape, "INT64")]
        inputs[0].set_data_from_numpy(input_ids)

        # Definisikan expected output
        outputs = [httpclient.InferRequestedOutput("logits")]

        # Execute inference
        response = self.client.infer(
            model_name=self.model_name,
            inputs=inputs,
            outputs=outputs,
        )
        return response.as_numpy("logits")

    def get_model_metadata(self) -> dict:
        metadata = self.client.get_model_metadata(self.model_name)
        return {
            "name": metadata["name"],
            "versions": metadata["versions"],
            "inputs": [{"name": i["name"], "shape": i["shape"]} for i in metadata["inputs"]],
        }

# Contoh penggunaan
client = TritonMLClient(url="localhost:8000", model_name="text_classifier")

# Single inference
input_ids = np.array([[101, 2023, 2003, 1037, 102] + [0]*123], dtype=np.int64)

print(f"Logits shape: {logits.shape}")

# Batch inference (Triton handle batching secara otomatis)

Dynamic Batching: pengaturan kritismax_queue_delay_microseconds adalah parameter yang paling berpengaruh. Nilai 5000 (5ms) berarti Triton akan menunggu hingga 5ms untuk mengumpulkan request sebelum dispatch batch. Untuk P99 SLA 100ms, nilai ini masih aman. Tapi untuk P99 SLA 20ms, 5ms delay sudah 25% budget latensi kamu — turunkan ke 1000-2000µs. Selalu set preferred_batch_size berdasarkan profiling GPU kamu, bukan default.

# model.py untuk Triton Python Backend (preprocessing pipeline)
import json
import numpy as np
import triton_python_backend_utils as pb_utils
from transformers import AutoTokenizer

class TritonPythonModel:
    """Python backend untuk tokenization — dijalankan sebagai step pertama ensemble."""

    def initialize(self, args):
        """Dipanggil sekali saat model load. Load berat resources di sini."""
        model_config = json.loads(args["model_config"])
        self.tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")
        self.max_length = 128

    def execute(self, requests):
        """Dipanggil untuk setiap (dynamic) batch request."""
        responses = []
        for request in requests:
            # Ambil input text dari request
            texts_tensor = pb_utils.get_input_tensor_by_name(request, "text")
            texts = [t.decode("utf-8") for t in texts_tensor.as_numpy().flatten()]

            # Tokenize
            enc = self.tokenizer(
                texts, max_length=self.max_length,
                padding="max_length", truncation=True,
                return_tensors="np"
            )

            # Buat output tensors
            out_ids = pb_utils.Tensor("input_ids", enc["input_ids"].astype(np.int64))
            out_mask = pb_utils.Tensor("attention_mask", enc["attention_mask"].astype(np.int64))
            responses.append(pb_utils.InferenceResponse([out_ids, out_mask]))
        return responses

    def finalize(self):
        """Cleanup saat model di-unload."""
        del self.tokenizer

2.4 Model Optimization // 10x lebih cepat, biaya 1/10

Model optimization untuk inference adalah salah satu area yang paling high-leverage di MLOps: investasi satu minggu engineer bisa menghasilkan 5–50x speedup yang langsung berdampak pada cost dan user experience. Ada tiga level optimasi yang bisa dilakukan secara komplementer: format conversion (ONNX, TorchScript), hardware-specific compilation (TensorRT, torch.compile), dan precision reduction (quantization INT8/FP16, pruning, knowledge distillation).

Penting untuk memahami mengapa optimasi ini bekerja. Model PyTorch saat inference masih mengandung banyak overhead dari framework dinamis: Python interpreter, autograd graph, dan general-purpose operator implementation. ONNX/TensorRT menghilangkan overhead ini dengan menyimpan computation graph yang sudah dioptimasi — operator fusion menggabungkan beberapa operasi (misal BatchNorm + ReLU) menjadi satu kernel GPU, mengurangi memory bandwidth dan kernel launch overhead.

Quantization INT8 lebih dari sekadar "FP32 → INT8". Prosesnya melibatkan calibration — mengamati distribusi aktivasi model pada representative dataset untuk menentukan scale factor yang tepat. Tanpa calibration yang baik, akurasi bisa turun drastis. Post-Training Quantization (PTQ) adalah titik awal yang cepat; Quantization-Aware Training (QAT) memberikan akurasi lebih baik dengan biaya training ulang.

Pythonmodel_optimization.py
import torch
import torch.nn as nn
import onnx
import onnxruntime as ort
import numpy as np

# === Step 1: Export ke ONNX ===
def export_to_onnx(model: nn.Module, save_path: str, input_shape=(1, 128)):
    model.eval()
    dummy_input = torch.randint(0, 30000, input_shape)
    torch.onnx.export(
        model,
        dummy_input,
        save_path,
        export_params=True,
        opset_version=17,         # gunakan opset terbaru untuk support operator terlengkap
        do_constant_folding=True, # fold constant ops — mengurangi graph size
        input_names=["input_ids"],
        output_names=["logits"],
        dynamic_axes={             # dynamic batch dan sequence length
            "input_ids": {0: "batch", 1: "seq_len"},
            "logits":    {0: "batch"},
        }
    )
    # Verifikasi ONNX graph valid
    onnx_model = onnx.load(save_path)
    onnx.checker.check_model(onnx_model)
    print(f"ONNX model saved dan verified: {save_path}")

# === Step 2: Run dengan ONNX Runtime ===
class ONNXPredictor:
    def __init__(self, model_path: str, use_gpu: bool = True):
        providers = ["CUDAExecutionProvider", "CPUExecutionProvider"] if use_gpu else ["CPUExecutionProvider"]
        sess_opts = ort.SessionOptions()
        sess_opts.graph_optimization_level = ort.GraphOptimizationLevel.ORT_ENABLE_ALL
        sess_opts.intra_op_num_threads = 4
        self.session = ort.InferenceSession(model_path, sess_opts, providers=providers)
        self.input_name = self.session.get_inputs()[0].name

    def predict(self, input_ids: np.ndarray) -> np.ndarray:
        return self.session.run(None, {self.input_name: input_ids})[0]

# === Step 3: ONNX → INT8 Quantization (Post-Training Quantization) ===
from onnxruntime.quantization import quantize_dynamic, QuantType

def quantize_model(input_path: str, output_path: str):
    quantize_dynamic(
        model_input=input_path,
        model_output=output_path,
        weight_type=QuantType.QInt8,        # weights jadi INT8
        optimize_model=True,
        per_channel=True,                    # lebih akurat dari per-tensor
        reduce_range=True,                   # gunakan 7-bit range untuk CPU compatibility
    )
    print(f"Quantized model saved: {output_path}")

# === Step 4: torch.compile — PyTorch 2.0 native optimization ===
def compile_model(model: nn.Module, mode: str = "reduce-overhead") -> nn.Module:
    """
    mode options:
    - "default": balance compile time vs runtime (bagus untuk eksperimen)
    - "reduce-overhead": agresif optimize, lebih lama compile, runtime lebih cepat
    - "max-autotune": paling agresif, butuh warmup banyak iteration
    """
    return torch.compile(model, mode=mode, fullgraph=True)
Validasi akurasi setelah setiap optimasiSetiap langkah optimasi berpotensi memperkenalkan numerik errors. Selalu jalankan benchmark akurasi end-to-end setelah ONNX export, setelah quantization, dan setelah TensorRT compilation. Untuk model klasifikasi, biasanya akurasi drop <0.5% untuk FP16 dan <1–2% untuk INT8 masih acceptable. Untuk model yang output digunakan dalam downstream computation (embedding, regression), validasi distribusi output secara kuantitatif, bukan hanya classification accuracy.

import tensorrt as trt
import numpy as np

class TensorRTConverter:
    """Convert ONNX model ke TensorRT engine untuk maksimum GPU performance."""

    def build_engine(
        self,
        onnx_path: str,
        engine_path: str,
        precision: str = "fp16",      # "fp32", "fp16", atau "int8"
        max_batch_size: int = 32,
    ):
        logger = trt.Logger(trt.Logger.WARNING)
        builder = trt.Builder(logger)
        network = builder.create_network(
            1 << int(trt.NetworkDefinitionCreationFlag.EXPLICIT_BATCH)
        )
        parser = trt.OnnxParser.OnnxParser(network, logger)

        with open(onnx_path, "rb") as f:
            assert parser.parse(f.read()), "ONNX parse failed"

        config = builder.create_builder_config()
        config.set_memory_pool_limit(trt.MemoryPoolType.WORKSPACE, 4 << 30)  # 4 GB workspace

        if precision == "fp16":
            config.set_flag(trt.BuilderFlag.FP16)
        elif precision == "int8":
            config.set_flag(trt.BuilderFlag.INT8)
            config.int8_calibrator = self._create_calibrator()  # WAJIB untuk INT8

        # Dynamic shape profiles
        profile = builder.create_optimization_profile()
        profile.set_shape(
            "input_ids",
            min=(1, 16),   max=(32, 128),  opt=(8, 64)
        )
        config.add_optimization_profile(profile)

        serialized_engine = builder.build_serialized_network(network, config)
        with open(engine_path, "wb") as f:
            f.write(serialized_engine)
        print(f"TensorRT engine ({precision}) saved: {engine_path}")

2.5 Serving Patterns // deploy tanpa downtime, rollout tanpa risiko

Cara kamu men-deploy model baru sama pentingnya dengan model itu sendiri. Sebuah model yang 5% lebih akurat tapi di-deploy dengan cara yang menyebabkan downtime atau degradasi untuk semua pengguna adalah jauh lebih buruk dari model lama yang stabil. Serving patterns adalah strategi deployment yang memungkinkan kamu memperbarui model dengan risiko minimal dan observability yang tinggi.

Blue-Green deployment adalah strategi paling sederhana: jalankan dua environment identik ("blue" = production saat ini, "green" = versi baru). Setelah green siap dan divalidasi, switch traffic 100% dari blue ke green secara instant. Rollback semudah switch balik. Kekurangan: butuh dua kali resource selama transisi.

Canary release lebih granular: kirim sebagian kecil traffic (misal 5%, lalu 20%, lalu 50%) ke versi baru secara bertahap. Jika metrik bagus, tingkatkan traffic. Jika ada degradasi, rollback hanya mempengaruhi sebagian pengguna. Ini adalah strategi standar untuk model ML karena kamu bisa validasi pada real traffic sebelum full rollout.

Shadow deployment adalah pola paling aman: model baru menerima copy traffic production tapi responnya tidak dikirim ke pengguna. Kamu bisa compare prediksi model baru vs model lama secara offline tanpa dampak apapun ke user. Ideal untuk validasi model sebelum canary.

A/B testing mirip canary tapi tujuannya berbeda: bukan untuk safe rollout, tapi untuk mengukur dampak bisnis dari dua versi model. Pengguna dibagi menjadi dua kelompok secara random dan konsisten (user-based splitting, bukan request-based) sehingga setiap user selalu mendapat model yang sama.

Pythonserving_patterns.py
import hashlib
from typing import Optional
from dataclasses import dataclass, field
from enum import Enum

class DeploymentStrategy(Enum):
    BLUE_GREEN = "blue_green"
    CANARY     = "canary"
    SHADOW     = "shadow"
    AB_TEST    = "ab_test"

@dataclass
class CanaryRouter:
    """Route traffic antara model lama dan baru berdasarkan canary_percentage."""
    stable_predictor: object
    canary_predictor: object
    canary_percentage: float = 0.05  # mulai dengan 5%

    def predict(self, features, request_id: str) -> dict:
        import random
        use_canary = random.random() < self.canary_percentage
        predictor = self.canary_predictor if use_canary else self.stable_predictor
        result = predictor.predict(features)
        result["model_variant"] = "canary" if use_canary else "stable"
        return result

class ShadowDeployment:
    """Jalankan shadow model secara async, bandingkan output tanpa mempengaruhi user."""
    def __init__(self, production_predictor, shadow_predictor, comparison_logger):
        self.prod = production_predictor
        self.shadow = shadow_predictor
        self.logger = comparison_logger

    async def predict(self, features) -> dict:
        import asyncio
        # Jalankan production dan shadow secara bersamaan
        prod_task = asyncio.create_task(self._run_async(self.prod, features))
        shadow_task = asyncio.create_task(self._run_async(self.shadow, features))

        prod_result = await prod_task
        # Shadow: fire and forget — tidak delay response ke user
        asyncio.ensure_future(self._log_shadow(shadow_task, prod_result))
        return prod_result  # hanya return production result ke user

    async def _log_shadow(self, shadow_task, prod_result):
        try:
            shadow_result = await shadow_task
            await self.logger.log_comparison(prod_result, shadow_result)
        except Exception as e:
            await self.logger.log_shadow_error(e)

class ABTestRouter:
    """User-consistent A/B split — user yang sama selalu dapat model yang sama."""
    def __init__(self, model_a, model_b, traffic_split: float = 0.5):
        self.model_a = model_a
        self.model_b = model_b
        self.traffic_split = traffic_split  # fraction ke model B

    def _get_variant(self, user_id: str) -> str:
        # Hash user_id → consistent assignment (bukan random per-request)
        h = int(hashlib.md5(user_id.encode()).hexdigest(), 16)
        return "B" if (h % 100) < int(self.traffic_split * 100) else "A"

    def predict(self, features, user_id: str) -> dict:
        variant = self._get_variant(user_id)
        model = self.model_b if variant == "B" else self.model_a
        result = model.predict(features)
        result["ab_variant"] = variant  # log ini untuk analysis!
        return result

Bisnis: conversion rate, click-through rate, user retention. Model bisa secara teknis "lebih akurat" di offline evaluation tapi menurunkan business metric di production — fenomena ini disebut metric misalignment dan sangat umum. from prometheus_client import Counter, Histogram, Gauge
import functools, time

# Prometheus metrics untuk monitoring serving patterns
inference_latency = Histogram(
    "model_inference_latency_seconds",
    "Inference latency distribution",
    labelnames=["model_version", "variant"],
    buckets=[.005, .01, .025, .05, .1, .25, .5, 1.0]
)
prediction_counter = Counter(
    "model_predictions_total",
    "Total predictions per variant",
    labelnames=["model_version", "variant", "label"]
)
error_counter = Counter(
    "model_errors_total",
    "Total prediction errors",
    labelnames=["model_version", "error_type"]
)

def monitor_prediction(model_version: str, variant: str = "stable"):
    """Decorator untuk auto-monitoring setiap prediction call."""
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            start = time.perf_counter()
            try:
                result = await func(*args, **kwargs)
                latency = time.perf_counter() - start
                inference_latency.labels(model_version, variant).observe(latency)
                prediction_counter.labels(model_version, variant, result["label"]).inc()
                return result
            except Exception as e:
                error_counter.labels(model_version, type(e).__name__).inc()
                raise
        return wrapper
    return decorator
