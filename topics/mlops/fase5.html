5.1 CI/CD untuk ML // automate everything

CI/CD untuk software tradisional sudah matang — setiap push trigger build, test, dan deploy otomatis. Tapi CI/CD untuk ML lebih kompleks: kita tidak hanya menguji kode, tapi juga model, data, dan perilaku sistem secara end-to-end. Pipeline ML yang baik harus bisa reproduce hasil yang sama, mendeteksi regresi model, dan deploy dengan zero-downtime — semua secara otomatis.

GitHub Actions adalah pilihan paling pragmatis untuk MLOps CI/CD karena terintegrasi langsung dengan repo, mendukung GPU runners (via self-hosted atau cloud runners), dan bisa trigger pipeline dari berbagai event: push, PR, schedule, atau workflow dispatch untuk manual trigger. Kombinasi dengan DVC untuk data versioning dan MLflow untuk experiment tracking menciptakan pipeline yang fully auditable.

ML CI/CD Pipeline — Dari Push ke Production
📝 Code Push
🧪 Lint & Unit Test
📊 Data Validation
🚀 Deploy (Canary)
🤖 Model Test
🏋️ Train & Eval
Setiap stage bisa fail-fast: data validation gagal → training tidak jalan. Model test gagal → deploy tidak terjadi.
YAML.github/workflows/ml-pipeline.yml
name: ML Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 2 * * 1'  # Weekly retraining Senin 02:00
  workflow_dispatch:  # Manual trigger dengan input
    inputs:
      force_retrain:
        type: boolean
        default: false

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Python
        uses: actions/setup-python@v5
        with: { python-version: '3.11' }
      - name: Install deps
        run: pip install -r requirements.txt
      - name: Lint
        run: ruff check . && mypy src/
      - name: Unit tests
        run: pytest tests/unit/ -v --tb=short
      - name: Validate data schema
        run: python scripts/validate_data.py

  train:
    needs: validate
    runs-on: [self-hosted, gpu]  # GPU runner untuk training
    steps:
      - uses: actions/checkout@v4
      - name: Pull data with DVC
        run: dvc pull
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
      - name: Train model
        run: python src/train.py --config configs/prod.yaml
      - name: Evaluate & compare
        run: python scripts/evaluate.py --challenger new --champion registry
      - name: Register if better
        run: python scripts/register_model.py --promote-if-better

  deploy:
    needs: train
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Deploy canary (10%)
        run: kubectl apply -f k8s/canary.yaml
      - name: Run smoke tests
        run: pytest tests/smoke/ --base-url ${{ vars.CANARY_URL }}
      - name: Promote to production
        run: kubectl apply -f k8s/production.yaml
Pythonscripts/evaluate.py
import mlflow
import sys
from mlflow.tracking import MlflowClient

def compare_models(challenger_run_id: str, champion_name: str) -> bool:
    """Return True jika challenger lebih baik dari champion."""
    client = MlflowClient()

    # Ambil metrics challenger dari run saat ini
    challenger = client.get_run(challenger_run_id)
    challenger_f1 = challenger.data.metrics['f1_score']
    challenger_latency = challenger.data.metrics['p99_latency_ms']

    # Ambil metrics champion dari production registry
    champion_version = client.get_model_version_by_alias(champion_name, 'production')
    champion_run = client.get_run(champion_version.run_id)
    champion_f1 = champion_run.data.metrics['f1_score']
    champion_latency = champion_run.data.metrics['p99_latency_ms']

    print(f"Champion: F1={champion_f1:.4f}, P99={champion_latency:.1f}ms")
    print(f"Challenger: F1={challenger_f1:.4f}, P99={challenger_latency:.1f}ms")

    # Challenger menang jika F1 lebih baik DAN latency tidak lebih buruk dari 10%
    f1_improved = challenger_f1 > champion_f1 * 1.001  # min +0.1%
    latency_ok = challenger_latency <= champion_latency * 1.10   # max +10%

    if f1_improved and latency_ok:
        print("✅ Challenger wins — promoting to staging")
        return True
    else:
        print("❌ Champion retained")
        return False

if __name__ == '__main__':
    winner = compare_models(
        challenger_run_id=os.environ['MLFLOW_RUN_ID'],
        champion_name='fraud-detector'
    )
    sys.exit(0 if winner else 1)  # Non-zero exit = pipeline gagal
Self-hosted vs Cloud Runners: GPU training di GitHub Actions membutuhkan self-hosted runner atau cloud runners berbayar (AWS CodeBuild, GCP Cloud Build). Untuk prototyping, pakai CPU-only runner dengan dataset kecil. Untuk production, dedicated GPU runner lebih cost-effective daripada cloud per-minute billing untuk training job yang panjang.
5.2 Model Testing Strategies // beyond accuracy metrics

Testing model ML sangat berbeda dari testing software biasa. Unit test untuk fungsi deterministik mudah — kamu tahu input dan expected output persis. Tapi model ML adalah probabilistic system: output bervariasi, "benar" tidak selalu binary, dan failure mode seringkali subtle dan distributional. Kamu perlu strategi testing berlapis.

Marco Ribeiro et al. (2020) memperkenalkan Behavioral Testing (CheckList) — framework yang terinspirasi dari software testing: Minimum Functionality Tests (MFT) untuk kemampuan dasar, Invariance Tests (INV) untuk robustness terhadap perturbasi yang seharusnya tidak mengubah prediksi, dan Directional Expectation Tests (DIR) untuk memastikan perturbasi tertentu mengubah output ke arah yang benar.

Pythontests/model/test_behavioral.py
import pytest
from src.model import SentimentClassifier

@pytest.fixture(scope='session')
def model():
    return SentimentClassifier.load_from_registry('production')

class TestMinimumFunctionality:
    """MFT: kemampuan dasar yang HARUS dimiliki model."""

    def test_obvious_positive(self, model):
        # Model harus benar untuk kalimat positif yang jelas
        obvious_positives = [
            "This product is absolutely amazing!",
            "I love this, best purchase ever",
            "Highly recommend to everyone",
        ]
        for text in obvious_positives:
            pred = model.predict(text)
            assert pred['label'] == 'POSITIVE', f"Failed on: {text}"
            assert pred['confidence'] > 0.85

    def test_obvious_negative(self, model):
        obvious_negatives = [
            "Terrible product, complete waste of money",
            "Worst experience I've ever had",
        ]
        for text in obvious_negatives:
            assert model.predict(text)['label'] == 'NEGATIVE'

class TestInvariance:
    """INV: perturbasi yang seharusnya TIDAK mengubah prediksi."""

    def test_typo_robustness(self, model):
        # Typo kecil tidak boleh mengubah sentiment
        original = "This product is great"
        with_typo = "Tis prodcut is graet"
        assert model.predict(original)['label'] == model.predict(with_typo)['label']

    def test_name_invariance(self, model):
        # Ganti nama orang tidak boleh mengubah sentiment
        text1 = "John loved the product"
        text2 = "Sarah loved the product"
        assert model.predict(text1)['label'] == model.predict(text2)['label']

class TestDirectional:
    """DIR: perturbasi yang HARUS mengubah prediksi ke arah tertentu."""

    def test_negation_flips_sentiment(self, model):
        # Menambah "not" harus mengubah arah sentiment
        positive = "I like this product"
        negated = "I do not like this product"
        pos_score = model.predict(positive)['positive_prob']
        neg_score = model.predict(negated)['positive_prob']
        assert neg_score < pos_score, "Negation should reduce positive probability"
Pythontests/data/test_schema.py
from great_expectations.dataset import PandasDataset
import pandas as pd

def validate_training_data(df: pd.DataFrame) -> dict:
    """Validasi schema dan distribusi data training sebelum dipakai."""
    ds = PandasDataset(df)

    results = {}

    # Schema checks
    results['columns_exist'] = ds.expect_table_columns_to_match_ordered_list(
        ['text', 'label', 'created_at']
    )['success']

    # Tidak ada null di kolom kritis
    results['no_null_text'] = ds.expect_column_values_to_not_be_null('text')['success']
    results['no_null_label'] = ds.expect_column_values_to_not_be_null('label')['success']

    # Label harus dari set yang valid
    results['valid_labels'] = ds.expect_column_values_to_be_in_set(
        'label', ['POSITIVE', 'NEGATIVE', 'NEUTRAL']
    )['success']

    # Class balance check — tidak ada class yang dominasi >80%
    label_dist = df['label'].value_counts(normalize=True)
    results['balanced_classes'] = label_dist.max() < 0.80

    # Text length reasonable
    results['text_length_ok'] = ds.expect_column_value_lengths_to_be_between(
        'text', min_value=5, max_value=512
    )['success']

    failed = [k for k, v in results.items() if not v]
    if failed:
        raise ValueError(f"Data validation failed: {failed}")

    return results
Accuracy saja tidak cukup: Model dengan 95% accuracy bisa tetap gagal di production jika tidak robust terhadap typo, tidak fair terhadap kelompok tertentu, atau berperilaku aneh di edge cases. Behavioral testing mengekspos failure mode yang tidak terlihat dari aggregate metrics.
5.3 Deployment Strategies // zero-downtime model rollout

Deploy model baru ke production adalah momen paling berisiko dalam ML lifecycle. Model bisa punya akurasi tinggi di offline evaluation tapi berperilaku buruk di production karena distribusi data yang berbeda, edge cases yang tidak ada di test set, atau interaksi dengan sistem lain yang tidak diperhitungkan. Deployment strategy yang baik meminimalkan blast radius jika sesuatu salah.

Ada empat strategi utama, masing-masing dengan tradeoff antara keamanan, kompleksitas, dan kecepatan feedback. Pilihan strategi bergantung pada seberapa kritikal service, seberapa yakin kamu dengan model baru, dan kapasitas tim untuk monitor.

🔵 Blue-Green Deployment
Blue (current): 100% traffic
Green (new): 0% traffic

Test green thoroughly
↓
Switch: green = 100%
blue = 0% (standby)

Rollback: switch back instantly
Zero-downtime: ✅
Risk: all-or-nothing switch
🐤 Canary Release
Step 1: new model = 5% traffic
         monitor 1 hour
Step 2: new model = 25% traffic
         monitor 2 hours
Step 3: new model = 50% traffic
Step 4: new model = 100%

Rollback: reduce to 0% instantly
Risk: limited blast radius
Best for: production ML models
👻 Shadow Deployment
Production model: serves users
Shadow model: receives same
  requests but responses DISCARDED

Compare outputs offline:
- agreement rate
- latency difference
- edge case discovery

Risk: zero (no user impact)
Cost: 2× compute

import asyncio
import httpx
from fastapi import FastAPI, Request

app = FastAPI()

async def shadow_request(url: str, payload: dict):
    """Fire-and-forget ke shadow model — response dibuang."""
    try:
        async with httpx.AsyncClient(timeout=5.0) as client:
            resp = await client.post(url, json=payload)
            # Log untuk comparison, bukan untuk serve ke user
            log_shadow_result(payload, resp.json())
    except Exception as e:
        # Shadow failure tidak boleh affect production
        logger.warning(f"Shadow request failed: {e}")

@app.post("/predict")
async def predict(request: Request):
    payload = await request.json()

    # Production response (yang di-serve ke user)
    prod_response = await call_production_model(payload)

    # Shadow request — async, tidak block production response
    asyncio.create_task(shadow_request(SHADOW_URL, payload))

    return prod_response  # Return production result immediately

5.4 Cost Optimization // ML bisa sangat mahal kalau tidak dikelola

Cost ML di production terdiri dari dua komponen besar: training cost (GPU compute, storage) dan inference cost (serving infrastructure). Training biasanya one-time atau periodic, tapi inference berjalan 24/7 — inilah yang mendominasi total cost untuk model yang mature. Optimasi inference bahkan perubahan kecil bisa berdampak besar: mengurangi latency 30% bisa berarti mengurangi jumlah instance 30% juga.

💰 ML Cost Breakdown — Training vs Inference
GPU instances: 4×
Monthly cost: $2,400
With INT8 quant: -40%
Pythonsrc/optimization/quantize.py
import torch
from torch.quantization import quantize_dynamic
from optimum.onnxruntime import ORTQuantizer, AutoQuantizationConfig

def quantize_for_inference(model_path: str, output_path: str, method: str = 'int8'):
    """
    Quantize model untuk inference yang lebih cepat dan hemat memory.
    INT8: ~4× smaller, ~2× faster, <1% accuracy drop
    FP16: ~2× smaller, ~1.5× faster, minimal accuracy drop
    """
    if method == 'dynamic_int8':
        # PyTorch dynamic quantization — paling mudah
        model = torch.load(model_path)
        quantized = quantize_dynamic(
            model,
            {torch.nn.Linear},  # Quantize hanya linear layers
            dtype=torch.qint8
        )
        torch.save(quantized, output_path)

    elif method == 'onnx_int8':
        # ONNX Runtime quantization — lebih optimal untuk serving
        quantizer = ORTQuantizer.from_pretrained(model_path)
        qconfig = AutoQuantizationConfig.avx512_vnni(
            is_static=False,   # Dynamic quant, tidak butuh calibration data
            per_channel=False
        )
        quantizer.quantize(
            quantization_config=qconfig,
            save_dir=output_path,
        )

    # Benchmark sebelum dan sesudah
    original_size = get_model_size_mb(model_path)
    quantized_size = get_model_size_mb(output_path)
    print(f"Size: {original_size:.0f}MB → {quantized_size:.0f}MB ({1-quantized_size/original_size:.0%} reduction)")

def benchmark_latency(model_path: str, n_warmup: int = 10, n_runs: int = 100):
    import time, numpy as np
    import onnxruntime as ort

    session = ort.InferenceSession(model_path, providers=['CPUExecutionProvider'])
    dummy_input = {'input_ids': np.ones((1,128), dtype=np.int64)}

    for _ in range(n_warmup): session.run(None, dummy_input)

    latencies = []
    for _ in range(n_runs):
        t0 = time.perf_counter()
        session.run(None, dummy_input)
        latencies.append((time.perf_counter()-t0)*1000)

    print(f"P50: {np.percentile(latencies,50):.1f}ms | P99: {np.percentile(latencies,99):.1f}ms")
Spot Instances untuk Training: AWS Spot / GCP Preemptible / Azure Spot bisa 60–90% lebih murah dari on-demand. Gunakan untuk training dengan checkpointing setiap epoch — jika instance di-preempt, resume dari checkpoint terakhir. Tidak cocok untuk inference (availability tidak terjamin), tapi sangat cocok untuk training dan batch inference.

5.5 MLOps Maturity & Production Checklist // dari chaos ke excellence

Google MLOps whitepaper mendefinisikan tiga level MLOps maturity yang menggambarkan seberapa otomatis dan scalable ML workflow sebuah organisasi. Tidak semua tim perlu mencapai level tertinggi — level yang tepat bergantung pada jumlah model di production, frekuensi retraining, dan ukuran tim ML. Yang penting: tahu di level mana kamu berada dan apa langkah selanjutnya.

🎯 MLOps Maturity Level Explorer
Pilih level:
Training manual di notebook. Deploy manual. Tidak ada monitoring. Experiment tidak reproducible. Kebanyakan tim ML ada di sini.

Start simple, then automate: Jangan langsung mengejar Level 2. Mulai dari Level 0 → Level 1 lebih dulu: pasang MLflow tracking, buat model registry, dan automated retraining script. Setelah itu stabil, baru bangun CI/CD pipeline lengkap. Premature MLOps complexity lebih berbahaya dari tidak ada MLOps sama sekali.
