3.1 Docker untuk ML // reproducible environments untuk model

Salah satu masalah paling frustasi dalam MLOps adalah environment hell: "works on my machine" tapi crash di server karena versi CUDA berbeda, dependency konflik, atau library yang tidak terinstall. Docker menyelesaikan masalah ini dengan meng-containerize seluruh environment — OS dependencies, Python packages, CUDA toolkit, hingga model weights — ke dalam sebuah image yang berjalan identik di mana saja.

Tapi Docker untuk ML bukan sekadar COPY . . lalu selesai. ML images punya tantangan unik: ukuran image yang bisa mencapai puluhan gigabyte (CUDA + PyTorch + model weights), kebutuhan GPU yang memerlukan nvidia-docker runtime, dan build time yang lama jika tidak di-cache dengan benar. Memahami multi-stage builds, layer caching strategy, dan GPU passthrough adalah skill fundamental yang akan menghemat jam kerja setiap minggu.

Empat prinsip Docker untuk ML yang sering dilanggar

  • Pin semua versi — pytorch==2.3.0, bukan pytorch. Sebuah minor version bump PyTorch bisa break model yang sudah di-deploy. Gunakan pip freeze > requirements.txt SETELAH testing, bukan sebelumnya.
  • Copy requirements sebelum kode — Layer Docker di-cache. Jika kamu copy kode dulu baru install requirements, setiap perubahan satu baris kode akan re-install semua packages. Urutkan: COPY requirements.txt → RUN pip install → COPY kode.
  • Pisahkan build env dari runtime env (multi-stage) — Build tools (gcc, cmake) tidak perlu ada di production image. Multi-stage builds bisa memangkas ukuran image 60–80%.
  • Jangan run sebagai root — Security best practice: buat non-root user, terutama untuk model serving. Banyak security scanner akan flag image yang run sebagai root.

import subprocess
import torch

def verify_gpu_setup():
    """Verifikasi GPU tersedia di dalam container."""
    # Cek CUDA dari PyTorch
    cuda_available = torch.cuda.is_available()
    device_count = torch.cuda.device_count()
    print(f"CUDA available: {cuda_available}")
    print(f"GPU count: {device_count}")

    if cuda_available:
        for i in range(device_count):
            props = torch.cuda.get_device_properties(i)
            vram_gb = props.total_memory / (1024**3)
            print(f"GPU {i}: {props.name} — {vram_gb:.1f} GB VRAM")

        # Test sederhana: alokasi tensor di GPU
        x = torch.randn(1000, 1000, device="cuda")
        y = x @ x.T  # matrix multiply
        print(f"GPU compute test OK: {y.shape}")

    # Cek nvidia-smi dari luar PyTorch
    result = subprocess.run(
        ["nvidia-smi", "--query-gpu=name,memory.total,driver_version",
         "--format=csv,noheader"],
        capture_output=True, text=True
    )
    if result.returncode == 0:
        print(f"nvidia-smi: {result.stdout.strip()}")

3.2 Kubernetes untuk ML // orkestrasi workload di skala

Docker menyelesaikan masalah packaging, tapi tidak menjawab pertanyaan: bagaimana menjalankan puluhan container model secara bersamaan, memastikan satu yang crash langsung digantikan, mendistribusikan traffic secara merata, dan mengelola resource GPU yang mahal secara efisien? Kubernetes (K8s) adalah jawabannya — sebuah container orchestration platform yang menjadi standar industri untuk menjalankan ML workload di production.

Untuk ML engineer, memahami tiga abstraksi K8s yang paling penting adalah: Pod (unit terkecil — satu atau beberapa container yang berjalan bersama), Deployment (deklarasi bagaimana Pod harus berjalan — berapa replika, image apa, resource apa), dan Service (abstraksi jaringan yang memungkinkan Pod lain atau user luar mengakses Pod kamu, meski Pod-nya restart dan IP-nya berubah). GPU scheduling di K8s membutuhkan NVIDIA Device Plugin dan resource request yang tepat.

GPU scheduling di K8s: mengapa limit == requestUntuk GPU, K8s mengharuskan limits.nvidia.com/gpu == requests.nvidia.com/gpu. Ini karena GPU adalah non-shareable resource — berbeda dengan CPU yang bisa di-throttle. Jika kamu request 1 GPU, kamu mendapat exclusive access ke GPU tersebut. Solusi untuk time-sharing GPU: NVIDIA MPS (Multi-Process Service) atau NVIDIA MIG (Multi-Instance GPU) untuk A100/H100 yang memungkinkan satu GPU fisik dipecah menjadi beberapa GPU logis.

3.3 Ray untuk Distributed ML // scale training dan serving ke banyak node

Ketika dataset atau model kamu terlalu besar untuk satu mesin, kamu butuh distributed computing. Ray adalah framework Python untuk distributed computing yang dirancang khusus untuk ML — bukan hanya data processing seperti Spark. Ray mengabstraksikan kompleksitas distribusi: kamu menulis kode Python biasa, Ray yang mengurus task scheduling, memory management, dan fault tolerance di cluster.

Ekosistem Ray mencakup empat komponen yang saling melengkapi: Ray Core (primitif distributed computing: remote functions dan actors), Ray Train (distributed training untuk PyTorch, TensorFlow, XGBoost), Ray Tune (distributed hyperparameter search), dan Ray Serve (scalable model serving). Keunggulan Ray dibanding alternatif: Pythonic, low overhead untuk task granularity yang kecil, dan integrasi deep dengan ML frameworks.

Pythonray_core_basics.py
import ray
import time
import numpy as np

# Inisialisasi Ray — connect ke cluster atau start local cluster
# ray.init(address='auto')  # connect ke cluster yang sudah ada
ray.init()  # local cluster untuk development

# ─── RAY CORE: Remote Functions ───────────────────────────────
# @ray.remote mengubah function biasa menjadi distributable task
# Return value-nya adalah ObjectRef — semacam "promise" ke hasil
@ray.remote
def evaluate_model(model_weights: np.ndarray, batch: np.ndarray) -> dict:
    """Jalankan inference di remote worker — tanpa blocking main thread."""
    # Simulasi inference time
    time.sleep(0.1)
    predictions = np.dot(batch, model_weights)
    return {"predictions": predictions.tolist(), "batch_size": len(batch)}

# Sequential: 10 batches × 0.1s = ~1 detik
weights = np.random.randn(128, 10)
batches = [np.random.randn(32, 128) for _ in range(10)]

# Parallel dengan Ray: semua batch dijalankan concurrently
# ray.put() upload data ke Ray object store sekali — tidak di-copy per task
weights_ref = ray.put(weights)

futures = [evaluate_model.remote(weights_ref, b) for b in batches]
results = ray.get(futures)  # blocking — tunggu semua selesai
print(f"Processed {sum(r['batch_size'] for r in results)} samples")

# ─── RAY ACTORS: Stateful Distributed Object ──────────────────
# Actor = class yang dijalankan di remote worker, menyimpan state
@ray.remote(num_gpus=1)  # request 1 GPU untuk actor ini
class ModelServer:
    def __init__(self, model_path: str):
        import torch
        self.model = torch.load(model_path, map_location="cuda")
        self.model.eval()
        self.request_count = 0

    def predict(self, inputs):
        import torch
        self.request_count += 1
        with torch.no_grad():
            return self.model(torch.tensor(inputs).cuda()).cpu().numpy()

    def get_stats(self):
        return {"request_count": self.request_count}

Pythonray_train_distributed.py
import ray
from ray import train
from ray.train.torch import TorchTrainer
from ray.train import ScalingConfig, RunConfig, CheckpointConfig
import torch
import torch.nn as nn

def train_func(config: dict):
    """Function ini dijalankan di SETIAP worker secara paralel."""
    # Prepare model — Ray Train otomatis wrap dengan DDP
    model = nn.Linear(128, config["num_classes"])
    model = train.torch.prepare_model(model)  # wrap DDP otomatis

    optimizer = torch.optim.Adam(model.parameters(), lr=config["lr"])
    criterion = nn.CrossEntropyLoss()

    # Prepare dataloader — otomatis di-shard per worker
    dataset = train.get_dataset_shard("train")
    loader = dataset.iter_torch_batches(
        batch_size=config["batch_size"], dtypes={"x": torch.float32, "y": torch.long}
    )

    for epoch in range(config["epochs"]):
        total_loss = 0.0
        for batch in loader:
            optimizer.zero_grad()
            outputs = model(batch["x"])
            loss = criterion(outputs, batch["y"])
            loss.backward()
            optimizer.step()
            total_loss += loss.item()

        # Report metrics ke Ray Train — untuk monitoring dan early stopping
        train.report({
            "loss": total_loss,
            "epoch": epoch,
        }, checkpoint=train.Checkpoint.from_dict({"model": model.state_dict()}))

# Konfigurasi training terdistribusi
trainer = TorchTrainer(
    train_loop_per_worker=train_func,
    train_loop_config={
        "lr": 1e-3,
        "epochs": 10,
        "batch_size": 64,
        "num_classes": 10,
    },
    scaling_config=ScalingConfig(
        num_workers=4,              # 4 GPU workers
        use_gpu=True,
        resources_per_worker={"GPU": 1, "CPU": 4},
    ),
    run_config=RunConfig(
        name="distributed-training-run",
        checkpoint_config=CheckpointConfig(
            num_to_keep=3,  # simpan 3 checkpoint terbaik
            checkpoint_score_attribute="loss",
            checkpoint_score_order="min",
        )
    )
)

result = trainer.fit()
print(f"Best checkpoint: {result.best_checkpoints}")

# ─── RAY TUNE: Distributed Hyperparameter Search ─────────────
from ray import tune
from ray.tune.schedulers import ASHAScheduler

tuner = tune.Tuner(
    trainable=TorchTrainer(train_loop_per_worker=train_func,
        scaling_config=ScalingConfig(num_workers=2, use_gpu=True)),
    param_space={
        "train_loop_config": {
            "lr": tune.loguniform(1e-5, 1e-2),   # log-uniform search
            "batch_size": tune.choice([32, 64, 128]),
            "epochs": 20,
            "num_classes": 10,
        }
    },
    tune_config=tune.TuneConfig(
        num_samples=20,   # 20 trial configurations
        scheduler=ASHAScheduler(   # early stop bad trials
            metric="loss", mode="min", max_t=20, grace_period=3),
    )
)

3.4 Feature Store // satu sumber kebenaran untuk fitur ML

Bayangkan dua tim ML di perusahaan yang sama, keduanya membangun model fraud detection. Tim A mendefinisikan fitur "rata-rata transaksi 7 hari" dengan cara tertentu, Tim B mendefinisikannya sedikit berbeda. Model di production menggunakan definisi yang berbeda dari yang digunakan saat training. Inilah tiga masalah utama yang Feature Store dirancang untuk diselesaikan: duplikasi komputasi fitur, training-serving skew, dan inkonsistensi definisi fitur antar tim.

Feature Store memiliki dua komponen utama: Offline Store (historical feature data untuk training — biasanya berbasis columnar storage seperti Parquet atau BigQuery) dan Online Store (low-latency feature lookup untuk serving — biasanya Redis atau DynamoDB). Konsep yang paling sering disalahpahami adalah point-in-time correctness: saat melatih model dengan data historis, fitur untuk setiap label harus diambil dari waktu label tersebut terjadi, bukan dari "sekarang" — karena feature values berubah seiring waktu.

Training-serving skew: silent killer model performanceTraining-serving skew terjadi ketika fitur yang digunakan saat training dihitung secara berbeda dari yang digunakan saat inference. Contoh: training menggunakan customer.avg_spend_7d dihitung dari data warehouse (lambat, batch), tapi serving menggunakan API endpoint yang ternyata menghitung dengan jendela waktu yang berbeda atau menggunakan timezone yang tidak konsisten. Model di A/B test bisa tampak bagus, tapi degradasi performance akibat skew sering baru ketahuan setelah berbulan-bulan.

# Feast: open-source feature store

from datetime import timedelta
import pandas as pd
from feast import (
    Entity, FeatureView, Field, FeatureStore,
    FileSource, RedisOnlineStoreConfig
)
from feast.types import Float32, Int64, String

# ─── 1. Definisi Entity (kunci lookup) ────────────────────────
customer = Entity(
    name="customer_id",
    join_keys=["customer_id"],
    description="Unique customer identifier"

# ─── 2. Feature Source (data mentah) ──────────────────────────
# Bisa berupa Parquet, BigQuery, Snowflake, Kafka (streaming)
customer_stats_source = FileSource(
    path="data/customer_stats.parquet",
    timestamp_field="event_timestamp",  # WAJIB untuk point-in-time
    created_timestamp_column="created_at",

# ─── 3. Feature View (definisi fitur + TTL) ───────────────────
customer_features_view = FeatureView(
    name="customer_features",
    entities=[customer],
    ttl=timedelta(days=30),  # fitur expire setelah 30 hari di online store
    schema=[
        Field(name="avg_transaction_7d",  dtype=Float32),
        Field(name="total_transactions_30d", dtype=Int64),
        Field(name="days_since_last_login", dtype=Int64),
        Field(name="preferred_category",    dtype=String),
    ],
    source=customer_stats_source,
    tags={"team": "ml-platform", "domain": "customer"},
)

# ─── 4. Materialisasi ke Online Store ─────────────────────────
store = FeatureStore(repo_path=".")

# Materialize: copy data dari offline ke online store (Redis)
# Biasanya dijalankan sebagai scheduled job (setiap jam / hari)
store.materialize_incremental(end_date=pd.Timestamp.now())

# ─── 5. Online Serving (low-latency, <10ms) ────────────────────
def get_features_for_serving(customer_ids: list) -> pd.DataFrame:
    """Ambil fitur terkini untuk inference — langsung dari Redis."""
    feature_vector = store.get_online_features(
        features=[
            "customer_features:avg_transaction_7d",
            "customer_features:total_transactions_30d",
            "customer_features:days_since_last_login",
        ],
        entity_rows=[{"customer_id": cid} for cid in customer_ids],
    )
    return feature_vector.to_df()

# Ini adalah fitur kritis Feature Store yang paling sering disalahpahami.
# Untuk setiap label (event_timestamp), Feast mengambil fitur
# dari waktu SEBELUM label tersebut terjadi — menghindari data leakage.

# Feast otomatis menangani point-in-time correctness!
training_df = store.get_historical_features(

# Feature freshness check — penting untuk monitoring
def check_feature_freshness(store, feature_view_name: str) -> dict:
    """Cek seberapa lama data di online store."""
    last_updated = store.get_feature_view(feature_view_name).last_updated_timestamp
    freshness_minutes = (pd.Timestamp.now() - last_updated).total_seconds() / 60
    return {
        "feature_view": feature_view_name,
        "last_updated": last_updated,
        "freshness_minutes": freshness_minutes,
        "is_stale": freshness_minutes > 60,  # alert jika > 1 jam
    }
Point-in-Time Correctness — Mengapa Urutan Waktu Sangat Penting
Skenario: prediksi churn. Feature avg_spend_7d untuk customer_id 1001
Timeline
Jan 1
Feature tersimpan: avg_spend_7d = 85.2
Jan 15 ← label
Training event: "customer churned" (label = 1)
Feast ambil fitur dari Jan 1 (sebelum Jan 15) → 85.2 ✓
Feb 1
Feature diupdate: avg_spend_7d = 210.5 (customer beli banyak)
❌ WRONG
Tanpa point-in-time: ambil fitur "sekarang" (Feb 1) = 210.5
→ Data leakage! Model belajar dari informasi yang belum ada saat churn

↑ Tanpa point-in-time correctness, model kamu training dengan "informasi dari masa depan" — yang tidak tersedia saat production. Hasilnya: model tampak hebat saat training, tapi jelek di production.

3.5 Cloud ML Platforms // managed vs self-hosted tradeoffs

Ketika memilih antara membangun infrastruktur ML sendiri vs menggunakan managed cloud platform, keputusan ini bukan soal mana yang "lebih baik" — tapi soal mana yang tepat untuk konteks spesifikmu. AWS SageMaker, GCP Vertex AI, dan Azure ML masing-masing menawarkan managed end-to-end ML platform: dari data preparation, training, experiment tracking, model registry, hingga deployment dan monitoring. Tapi mereka datang dengan tradeoff signifikan.

Argumen terkuat untuk managed platform: kamu tidak perlu mengelola infrastruktur Kubernetes, storage, networking, dan security dari nol. Engineer bisa fokus ke ML, bukan DevOps. Argumen terkuat untuk self-hosted: vendor lock-in yang serius (pipeline SageMaker tidak mudah dipindahkan ke Azure), biaya yang bisa 3–5x lebih mahal dibanding raw infrastructure, dan kurangnya fleksibilitas untuk use case non-standar. Rule of thumb: mulai dengan managed platform, pindah ke self-hosted setelah kamu tahu persis apa yang kamu butuhkan.

AWS SageMaker

Platform paling mature dan feature-complete. Kekuatan: integrasi tight dengan AWS ekosistem (S3, IAM, VPC), SageMaker Pipelines untuk MLOps workflow, built-in algorithms yang sangat dioptimasi untuk distributed training, SageMaker Inference untuk berbagai deployment pattern (real-time, batch, multi-model endpoint). Kelemahan: pricing kompleks dan mahal, UI yang overwhelming, learning curve tinggi untuk pemula.

GCP Vertex AI

Generasi kedua Google Cloud ML platform (menggantikan AI Platform yang deprecated). Kekuatan: integrasi native dengan BigQuery untuk feature engineering, Vertex AI Pipelines berbasis Kubeflow (open standard), excellent AutoML untuk non-expert. Kelemahan: ekosistem lebih kecil dari AWS, documentation kurang lengkap untuk edge cases, fitur masih lebih sedikit dari SageMaker di beberapa area.

Azure ML

Pilihan natural untuk enterprise yang sudah heavily invested di Microsoft stack (Active Directory, Azure DevOps, Power BI). Kekuatan: integrasi seamless dengan Azure DevOps untuk CI/CD, responsible AI tools paling mature (Fairlearn, InterpretML), enterprise governance yang kuat. Kelemahan: GPU availability kadang lebih terbatas dari AWS/GCP, UI kurang intuitif, beberapa fitur masih dalam preview.

Kapan Self-Hosted?

Self-hosted (Kubernetes + MLflow + Seldon/KServe + Feast) masuk akal ketika: (1) regulasi atau data sovereignty mengharuskan on-premise, (2) scale sudah cukup besar untuk justify engineering cost (biasanya >50 model di production), (3) kebutuhan customization yang tidak bisa dipenuhi managed platform, atau (4) vendor lock-in adalah risiko bisnis yang tidak bisa diterima. Bukan untuk startup early-stage.

Pythonsagemaker_training.py
import sagemaker
from sagemaker.pytorch import PyTorch
from sagemaker.inputs import TrainingInput
from sagemaker.debugger import ProfilerConfig, FrameworkProfile

# SageMaker session dan role dari IAM
session = sagemaker.Session()
role = sagemaker.get_execution_role()
bucket = session.default_bucket()

# ─── Definisi Training Job ─────────────────────────────────────
estimator = PyTorch(
    entry_point="train.py",         # training script kamu
    source_dir="./src",             # folder yang di-upload ke S3
    role=role,
    framework_version="2.3",
    py_version="py311",

    # Instance type — SageMaker punya ratusan opsi
    instance_type="ml.p3.8xlarge",  # 4x V100 GPU
    instance_count=2,               # 2 nodes × 4 GPU = 8 GPU total

    # Hyperparameters — bisa di-override oleh SageMaker HPO
    hyperparameters={
        "epochs": 50,
        "batch-size": 128,
        "learning-rate": 0.001,
        "model-name": "resnet50",
    },

    # SageMaker Profiler — otomatis capture GPU/CPU utilization
    profiler_config=ProfilerConfig(
        system_monitor_interval_millis=500,
        framework_profile_params=FrameworkProfile(num_steps=10)
    ),

    # Distributed training dengan SageMaker Data Parallel
    distribution={
        "smdistributed": {"dataparallel": {"enabled": True}}
    },

    # Checkpoint ke S3 untuk fault tolerance
    checkpoint_s3_uri=f"s3://{bucket}/checkpoints/run-001",

    # Spot training: bisa hemat 70–90% biaya, tapi bisa diinterupsi
    use_spot_instances=True,
    max_wait=7200,  # max 2 jam termasuk waktu tunggu spot
    max_run=3600,   # max 1 jam actual training
)

# Data channels — SageMaker download dari S3 ke /opt/ml/input/data/
estimator.fit({
    "train": TrainingInput(
        s3_data=f"s3://{bucket}/datasets/imagenet/train",
        content_type="application/x-recordio",
        input_mode="FastFile",  # stream dari S3, tidak perlu download penuh
    ),
    "validation": TrainingInput(
        s3_data=f"s3://{bucket}/datasets/imagenet/val",
        input_mode="FastFile",
    )
}, wait=False)  # async — tidak block Python process

print(f"Training job: {estimator.latest_training_job.name}")
