5.1 Bedah Arsitektur OpenClaw / Claude Code // belajar dari yang terbaik

OpenClaw dan Claude Code adalah contoh production coding agent yang sophisticated. Mereka bukan sekedar "LLM + tools" — ada banyak engineering decisions matang yang membuat mereka reliable, aman, dan powerful di kondisi nyata.

Sandboxed Execution Environment

Setiap shell command dieksekusi dalam environment terisolasi — container (Docker), subprocess dengan limited permissions (seccomp, Linux namespaces), atau VM. Agent tidak bisa merusak sistem host. Root cause: LLM bisa generate command berbahaya — sandbox adalah last line of defense.

Intelligent File Context

Agent tidak baca semua file sekaligus. Dimulai dengan structure overview (find, ls, tree), selective read berdasarkan relevance. File besar di-truncate dengan metadata (total lines, encoding). Kunci: agent tahu apa yang ada tanpa overwhelm context window.

Diff-Based Editing (str_replace)

Daripada rewrite file penuh, agent membuat minimal diffs via search-and-replace. Lebih efisien dari sisi token, mengurangi risiko unintended changes, dan memudahkan review. str_replace gagal jika string tidak ditemukan exact — ini intentional sebagai safety check.

Git Integration

Agent bisa membuat commits, branches, dan stashes. Ini memberikan natural checkpoint dan rollback capability. Jika agent menyebabkan masalah, git reset mengembalikan state. Juga memungkinkan feature branch per task.

Test Runner Loop

Setelah membuat perubahan, agent otomatis run tests dan iterate berdasarkan hasil. Loop: code change → run tests → analyze failures → fix → repeat. Ini self-verification yang sangat powerful — agent tidak berhenti sampai tests hijau.

Minimal Footprint Permission Model

User secara eksplisit approve scope: direktori mana yang bisa diakses, command apa yang bisa dijalankan, apakah bisa akses internet, dll. Agent selalu request minimum permission yang dibutuhkan, bukan maximum yang mungkin diperlukan.

import subprocess, os, pathlib

class BashTool:
    """Safe bash execution dengan workspace scoping."""
    def __init__(self, workspace: str, allowed_cmds: list[str] = None):
        self.workspace = os.path.realpath(workspace)
        self.allowed_cmds = allowed_cmds  # None = allow all (for dev)

    def __call__(self, command: str, timeout: int = 30) -> dict:
        if self.allowed_cmds:
            cmd_base = command.split()[0]
            if cmd_base not in self.allowed_cmds:
                return {"error": f"Command '{cmd_base}' not in allowlist"}
        try:
            result = subprocess.run(
                command, shell=True, capture_output=True, text=True,
                timeout=timeout, cwd=self.workspace,
                env={**os.environ, "HOME": self.workspace}
            )
            stdout = result.stdout[:50000]  # Truncate panjang output
            return {
                "stdout": stdout,
                "stderr": result.stderr[:5000],
                "returncode": result.returncode,
                "truncated": len(result.stdout) > 50000
            }
        except subprocess.TimeoutExpired:
            return {"error": f"Timed out after {timeout}s"}

class StrReplaceTool:
    """Minimal diff editing — safer than full file rewrite."""
    def __init__(self, workspace: str):
        self.workspace = workspace

    def __call__(self, path: str, old_str: str, new_str: str) -> dict:
        full_path = os.path.join(self.workspace, path.lstrip("/"))
        if not os.path.exists(full_path):
            return {"error": f"File not found: {path}"}
        content = open(full_path).read()
        count = content.count(old_str)
        if count == 0:
            return {"error": "old_str not found. Check exact whitespace and indentation."}
        if count > 1:
            return {"error": f"old_str found {count} times. Make it more specific."}
        open(full_path, "w").write(content.replace(old_str, new_str, 1))
        return {"success": True, "file": path}

5.2 Computer Use Agent // agent yang bisa melihat dan mengklik

Computer use agent dapat mengontrol komputer layaknya manusia: melihat screenshot, menggerakkan mouse, mengetik, dan berinteraksi dengan aplikasi apa pun. Ini membuka automasi yang tidak mungkin dengan tools berbasis API.

Screenshot-vision-action loop

1. Screenshot — ambil screenshot layar saat ini, encode sebagai base64, kirim ke model bersama task description.

2. Vision Analysis — model menganalisis screenshot: apa yang terlihat, di mana elemen UI berada, apa yang perlu diklik atau diketik.

3. Action Generation — model menghasilkan action object: mouse_move, left_click, right_click, double_click, type, key, scroll.

4. Execution — harness mengeksekusi action menggunakan pyautogui, xdotool, atau platform-native APIs.

5. Repeat — ambil screenshot baru (state baru setelah action) dan ulangi loop sampai task selesai.

Pythoncomputer_use_agent.py
import anthropic, pyautogui, base64
from io import BytesIO

class ComputerUseAgent:
    def __init__(self, display: tuple[int, int] = (1280, 800)):
        self.client = anthropic.Anthropic()
        self.display = display
        pyautogui.PAUSE = 0.5    # Safety pause antara actions
        pyautogui.FAILSAFE = True # Move ke corner untuk kill agent

    def screenshot(self) -> str:
        img = pyautogui.screenshot()
        img = img.resize(self.display)
        buf = BytesIO()
        img.save(buf, format="PNG", optimize=True)
        return base64.b64encode(buf.getvalue()).decode()

    def execute_action(self, action: dict):
        t = action["type"]
        if t == "mouse_move":
            pyautogui.moveTo(action["x"], action["y"], duration=0.3)
        elif t == "left_click":
            pyautogui.click(action["x"], action["y"])
        elif t == "double_click":
            pyautogui.doubleClick(action["x"], action["y"])
        elif t == "right_click":
            pyautogui.rightClick(action["x"], action["y"])
        elif t == "type":
            pyautogui.typewrite(action["text"], interval=0.05)
        elif t == "key":
            pyautogui.hotkey(*action["key"].split("+"))   # e.g. "ctrl+c"
        elif t == "scroll":
            pyautogui.scroll(action.get("direction", 3), x=action["x"], y=action["y"])
        elif t == "screenshot":
            pass   # Handled at loop level

    async def run(self, task: str, max_steps: int = 50) -> str:
        messages = []
        for step in range(max_steps):
            # Build message dengan screenshot saat ini
            content = [
                {"type": "image", "source": {
                    "type": "base64", "media_type": "image/png",
                    "data": self.screenshot()}},
                {"type": "text", "text": f"Task: {task}\nStep {step+1}/{max_steps}"}
            ]
            messages = [{"role": "user", "content": content}]

            response = self.client.messages.create(
                model="claude-opus-4-6",
                max_tokens=1024,
                tools=[{
                    "type": "computer_20241022",
                    "name": "computer",
                    "display_width_px": self.display[0],
                    "display_height_px": self.display[1],
                    "display_number": 1
                }],
                messages=messages
            )

            if response.stop_reason == "end_turn":
                texts = [b for b in response.content if b.type == "text"]
                return texts[0].text if texts else "Task completed"

            for block in response.content:
                if block.type == "tool_use":
                    self.execute_action(block.input)

        return "Max steps reached"

Production: Docker + X11 forwarding atau headless browserpyautogui cocok untuk pembelajaran dan prototyping di mesin lokal. Di production, computer use modern dieksekusi di dalam Docker container dengan X11 forwarding (virtual display via Xvfb) untuk isolasi mutlak dan latensi lebih rendah tanpa overhead VM penuh. Alternatif yang makin populer: gunakan headless browser (Playwright/Puppeteer) yang diinjeksi dengan visi — agent menerima screenshot dari browser, lalu mengeksekusi klik/ketik via Playwright API, bukan pyautogui. Pendekatan ini lebih deterministik, lebih mudah di-scale, dan tidak perlu display server.
5.3 Coding Agent Architecture // agent yang bisa koding seperti senior engineer

Coding agent yang baik bukan hanya "LLM yang bisa tulis kode" — ia memahami konteks codebase, membuat perubahan yang minimal dan tepat, menjalankan tests, dan iterate sampai selesai. Arsitekturnya punya beberapa fase yang distinct.

1
Explore Phase
Sebelum menulis kode, agent memahami codebase: structure, conventions, relevant files, existing patterns. Gunakan glob, grep, dan selective read.
2
Plan Phase
Buat rencana eksplisit: file mana yang perlu diubah, perubahan apa, dependencies yang perlu diinstall, tests yang perlu ditambah.
3
Implement Phase
Eksekusi plan menggunakan str_replace untuk edit minimal, write_file untuk file baru. Verifikasi setiap perubahan dengan read kembali.
4
Verify Phase
Run tests, linting, type checking. Jika gagal, analyze error dan kembali ke implement phase. Tidak berhenti sampai tests hijau.
5
Cleanup Phase
Remove debug code, update docs jika perlu, commit dengan pesan yang deskriptif. Review diff akhir.
SYSTEM PROMPTcoding_agent_system_prompt.txt
You are an expert software engineer assistant. Your job is to implement
changes to a codebase based on the user's requirements.

## Mandatory Workflow
1. EXPLORE first: Use glob/grep/read_file to understand the codebase
   structure before making any changes. Never modify files you haven't read.
2. PLAN explicitly: Before coding, state what you'll change and why.
   Consider dependencies, side effects, and tests.
3. IMPLEMENT minimally: Use str_replace for edits (not full rewrites).
   Make small, focused changes. One concern per tool call.
4. VERIFY constantly: Run tests after every meaningful change.
   If tests fail, analyze the error carefully before trying a fix.
5. STOP when: all tests pass AND the implementation satisfies the requirements.

## Engineering Rules
- Read before editing — never guess at file contents
- Follow existing code style, naming conventions, and patterns
- Prefer str_replace over write_file for existing files
- If stuck 3+ iterations on same error: step back and rethink approach
- Never delete code without understanding what it does
- Commit frequently with descriptive messages

## Error Handling Philosophy
When a test fails, analyze the FULL error message before attempting a fix.
Common mistakes: incorrect import paths, wrong function signatures, missing
dependencies, misunderstood existing API. Most fixes are small targeted changes,
not full rewrites.

5.5 Voice & Multimodal Agent // agent yang mendengar, melihat, dan berbicara

Generasi berikutnya agent bukan hanya teks — mereka menerima audio, gambar, video, dan merespons lewat suara secara real-time. Voice agent sudah production-ready di 2026 dengan latency sub-1-detik, membuka use case baru: customer service, aksesibilitas, dan interface yang lebih natural dari keyboard.

Tiga arsitektur multimodal agent

  • Cascaded (STT → LLM → TTS) — audio dikonversi ke teks (Whisper), diproses LLM, response dikonversi kembali ke audio (ElevenLabs/TTS). Latency ~2–4 detik. Mudah di-debug tapi terasa tidak natural
  • Native audio model — model menerima audio langsung dan output audio (GPT-4o Realtime, Gemini Live). Latency <1 detik. Lebih natural tapi lebih mahal dan kurang kontrol
  • Vision + action agent — menerima screenshot/kamera, reasoning visual, eksekusi klik/ketik (Computer Use). Tidak butuh API khusus — cukup Claude dengan tool screenshot

Tantangan unik voice agent

  • Interruption handling — user mungkin memotong agent di tengah berbicara. Sistem harus bisa stop playback dan mulai dengar lagi (VAD — Voice Activity Detection)
  • Response length calibration — system prompt harus secara eksplisit minta response singkat. LLM yang biasa menulis panjang akan membuat user bosan mendengar
  • Latency budget — STT ~300ms + LLM ~500ms + TTS ~400ms = ~1.2 detik minimum. Gunakan streaming di setiap layer untuk mengurangi latency persepsi
  • Multimodal context — kalau agent menerima gambar, pastikan model yang digunakan support vision (Claude claude-opus-4-8, GPT-4o). Gambar = token yang banyak, atur budget dengan ketat
2026: voice agent makin mudahLiveKit, Daily.co, dan Vapi.ai menyediakan infrastruktur WebRTC + VAD + STT/TTS siap pakai — kamu tinggal plug-in LLM-mu. Untuk prototype cepat, mulai dari sini sebelum bangun pipeline sendiri.
5.6 VLM sebagai Reasoning Engine // model yang melihat dan berpikir CV Multimodal

Vision-Language Model (VLM) bukan hanya "model yang bisa lihat gambar" — ketika dipakai sebagai reasoning engine sebuah agent, VLM memungkinkan agent untuk memahami dunia visual secara mendalam: membaca diagram, menganalisis screenshot UI, menginterpretasi grafik data, dan merespons terhadap perubahan visual di environment. Ini membuka kelas use case yang tidak mungkin dengan teks saja.

Model seperti Claude claude-opus-4-8, GPT-4o, dan Gemini 1.5 Pro sudah mature untuk production vision reasoning. Open-source alternatif: LLaVA, InternVL2, Qwen-VL — bisa dijalankan lokal via Ollama atau vLLM.

VLM vs LLM sebagai agent brain

LLM murni bergantung pada deskripsi teks dari state dunia. VLM menerima gambar langsung — tidak ada information loss dari konversi visual → teks. Untuk domain seperti UI automation, medical imaging, dan analisis dokumen visual, ini bukan sekadar nyaman: akurasinya secara fundamental lebih tinggi.

Token budget untuk gambar

Gambar mengkonsumsi token jauh lebih banyak daripada teks setara. Claude menghitung gambar berdasarkan resolusi; GPT-4o menggunakan tile system. Aturan praktis: resize gambar ke resolusi terendah yang masih cukup untuk task, gunakan detail: "low" untuk overview dan detail: "high" hanya ketika detail penting. Untuk agent yang loop banyak, ini bisa beda 10x dalam cost.

Grounding: dari piksel ke aksi

VLM yang baik bisa melakukan grounding — menghubungkan instruksi teks dengan lokasi spesifik di gambar. "Klik tombol Submit" membutuhkan model yang bisa identify koordinat pixel tombol tersebut. Model khusus seperti SeeClick dan Qwen-VL dioptimasi untuk grounding tugas UI.

Multimodal memory

Agent dengan VLM bisa menyimpan "visual memory" — screenshot state sebelumnya, diagram arsitektur, atau referensi visual. Ini penting untuk task yang membutuhkan konsistensi visual lintas banyak langkah, seperti form filling kompleks atau visual QA atas dokumen panjang.

Pythonvlm_reasoning_agent.py
import anthropic, base64, httpx
from pathlib import Path

client = anthropic.Anthropic()

def encode_image(path_or_url: str) -> dict:
    """Encode gambar lokal atau URL sebagai Anthropic image block."""
    if path_or_url.startswith("http"):
        data = httpx.get(path_or_url).content
        media_type = "image/jpeg"
    else:
        data = Path(path_or_url).read_bytes()
        ext = Path(path_or_url).suffix.lower()
        media_type = {".png":"image/png",".jpg":"image/jpeg",".webp":"image/webp"}.get(ext,"image/png")
    return {
        "type": "image",
        "source": {"type":"base64", "media_type":media_type,
                   "data":base64.b64encode(data).decode()}
    }

def vlm_agent_step(image_path: str, task: str, history: list) -> str:
    """Satu langkah reasoning VLM: lihat gambar → analisis → putuskan aksi."""
    content = [
        encode_image(image_path),
        {"type": "text", "text": f"Task: {task}\n\nAnalisis gambar ini dan tentukan aksi selanjutnya."}
    ]

    messages = history + [{"role": "user", "content": content}]

    response = client.messages.create(
        model="claude-opus-4-8",
        max_tokens=1024,
        system="""Kamu adalah visual reasoning agent. Untuk setiap gambar:
1. Deskripsikan apa yang kamu lihat secara akurat (UI elements, teks, layout)
2. Identifikasi elemen yang relevan dengan task
3. Tentukan aksi spesifik: klik koordinat (x,y), ketik teks, atau scroll
4. Jika task selesai, jawab dengan DONE: 

Format aksi: ACTION: {"type": "click", "x": 123, "y": 456}
              ACTION: {"type": "type", "text": "hello"}
              ACTION: {"type": "scroll", "direction": "down", "amount": 3}""",
        messages=messages
    )

    return response.content[0].text

# Contoh: agent menganalisis screenshot dan memutuskan aksi
if __name__ == "__main__":
    result = vlm_agent_step(
        image_path="screenshot.png",
        task="Isi form login dengan username 'demo' dan klik tombol Submit",
        history=[]
    )
    print(result)

5.7 Multimodal Tool Calling // agent yang memanggil vision sebagai tool LM CV Multimodal

Pola paling fleksibel untuk multimodal agent: LLM teks menjadi orkestrator, dan model vision dipanggil sebagai tool. Agent memutuskan kapan perlu melihat gambar, memanggil vision tool, menerima hasilnya sebagai teks terstruktur, lalu melanjutkan reasoning. Ini memisahkan concerns dengan bersih — orchestrator tidak harus VLM, dan vision model bisa diganti tanpa mengubah arsitektur agent.

Tiga pola multimodal tool calling

  • Vision-as-tool — agent teks memanggil analyze_image(path, question) dan menerima deskripsi terstruktur sebagai tool result. Paling fleksibel, bisa pakai model vision berbeda per task.
  • Natively multimodal agent — agent langsung terima gambar sebagai input message, reasoning visual dilakukan inline. Lebih efisien token tapi kurang modular.
  • Specialized vision subagent — orchestrator mendelegasikan seluruh task visual ke subagent khusus yang equipped dengan vision tools, menerima hasil terstruktur. Cocok untuk complex vision pipeline.

        system=system,
        messages=[{"role": "user", "content": [
            {"type":"image","source":{"type":"base64","media_type":"image/png","data":img_data}},
            {"type":"text","text":prompt}
        ]}]
    )
    return resp.content[0].text

def run_multimodal_agent(task: str, max_turns: int = 10) -> str:
    """Agent teks yang memanggil vision tools saat dibutuhkan."""
    messages = [{"role": "user", "content": task}]

    for _ in range(max_turns):
        resp = client.messages.create(
            model="claude-opus-4-8", max_tokens=2048, tools=TOOLS, messages=messages
        )

        if resp.stop_reason == "end_turn":
            return next(b.text for b in resp.content if b.type == "text")

        messages.append({"role": "assistant", "content": resp.content})
        tool_results = []
        for block in resp.content:
            if block.type == "tool_use":
                result = execute_vision_tool(block.name, block.input)
                tool_results.append({"type":"tool_result","tool_use_id":block.id,"content":result})

        messages.append({"role": "user", "content": tool_results})

    return "Max turns reached"

# Contoh penggunaan
result = run_multimodal_agent(
    "Saya punya invoice di 'invoice.png'. Ekstrak total amount, tanggal, dan nomor invoice-nya."
)
print(result)

Kapan pakai mana

  • Vision-as-tool — ketika agent punya banyak tools lain (search, database, API) dan vision hanya salah satunya. Paling maintainable.
  • Natively multimodal — ketika task-nya visual-first (analisis dokumen, UI testing) dan teks hanya konteks pendukung.
  • Vision subagent — ketika vision pipeline cukup kompleks untuk dipisah: multiple images, banyak pertanyaan berbeda, atau butuh retry logic terpisah.
5.8 Vision Pipeline Orchestration // agent yang menggabungkan model CV secara sequential CV Multimodal

Vision pipeline orchestration adalah pola di mana agent mengorkestrasikan beberapa model CV secara sequential atau paralel untuk menyelesaikan task yang lebih kompleks dari yang bisa dilakukan satu model saja. Contoh klasik: detect → crop → classify → caption — setiap langkah adalah model berbeda dengan output yang menjadi input langkah berikutnya. Agent memutuskan alur, menangani error, dan merakit hasil akhir.

1
Detect
Object detection (YOLO, DETR) — temukan semua objek relevan dalam gambar, dapatkan bounding boxes.
2
Crop & Preprocess
Crop region of interest dari bounding box, resize ke ukuran yang dibutuhkan model berikutnya.
3
Classify / Segment
Klasifikasi tiap crop (ResNet, EfficientNet) atau segmentasi pixel-level (SAM, Mask R-CNN).
4
Caption / Reason
VLM menghasilkan deskripsi natural language atau menjawab pertanyaan tentang crop yang sudah terklasifikasi.
5
Synthesize
Agent merakit semua hasil menjadi output final: laporan terstruktur, decision, atau aksi downstream.

def run_vision_pipeline_agent(image_path: str, task: str) -> dict:

    Dia memutuskan tool mana yang dipanggil dan dalam urutan apa.

        "role": "user",
        "content": f"Gambar: {image_path}\nTask: {task}"

            messages=messages

            try:
                return json.loads(text)
            except:

        messages.append({"role": "assistant", "content": resp.content})
        tool_results = []
        for block in resp.content:
            if block.type == "tool_use":

        messages.append({"role": "user", "content": tool_results})

Pola paralel vs sequential

  • Sequential — setiap langkah butuh output dari langkah sebelumnya. Contoh: detect dulu → crop → classify crop. Lebih mudah di-debug, latency adalah jumlah semua langkah.
  • Paralel (fan-out) — ketika banyak objek terdeteksi dan setiap objek bisa dianalisis independen. Kirim semua crops ke classifier secara bersamaan dengan asyncio.gather(). Latency = langkah terlama, bukan jumlah semua langkah.
  • Conditional branching — agent memutuskan langkah berikutnya berdasarkan hasil sebelumnya. Contoh: kalau objek adalah teks → gunakan OCR, kalau objek adalah wajah → gunakan face analysis. Agent yang fleksibel untuk dataset yang heterogen.
Error propagation di vision pipelineSetiap langkah bisa gagal: detection miss, segmentation tidak akurat, atau VLM hallucinate. Implement confidence threshold di setiap langkah — jika confidence di bawah threshold, agent harus bisa fallback (coba model alternatif, minta human review, atau skip langkah itu). Jangan biarkan error satu langkah cascade tanpa pengecekan.
5.9 Checklist Keahlian Agent Engineer // kapan kamu bisa disebut expert

Ini bukan checklist yang harus diselesaikan secara linear. Tapi kalau kamu bisa meng-check semua items ini dengan jujur, kamu memang sudah di level yang bisa disebut production agent engineer.

Fondasi (Fase 1–2)

  • Bisa jelaskan bagaimana LLM memutuskan kapan call tool vs jawab langsung, dan apa pengaruh temperature terhadap keputusan ini
  • Bisa implement agent harness minimal dari scratch dalam <100 baris Python tanpa framework
  • Paham perbedaan 4 jenis memory dan bisa pilih strategi yang tepat untuk use case spesifik
  • Bisa tulis tool schema — deskripsi, argumen, return format — yang LLM hampir tidak pernah salah interpretasi
  • Bisa implement context management untuk agent yang run lebih dari 1 jam tanpa crash karena overflow
  • Paham cara kerja KV cache dan prompt caching, serta bisa mengaktifkan caching untuk hemat 60–90% cost di agent yang loop banyak
  • Bisa hitung token budget secara proaktif sebelum setiap API call dan implement compression strategy sebelum context overflow
  • Bisa tulis system prompt yang robust: ada mandatory workflow, tool rules, error behavior, dan negative examples
  • Bisa implement RAG pipeline dari scratch: chunking, embedding, indexing, retrieval, dan inject hasil ke context agent
  • Bisa build checkpointing system sehingga agent yang berjalan lama bisa resume dari titik terakhir saat crash

Multi-Agent (Fase 3)

  • Bisa design orchestrator-subagent system dengan proper error propagation dari subagent ke orchestrator
  • Bisa implement fan-out/fan-in parallelism dengan rate limiting yang tidak blow up API quota
  • Bisa build dan deploy MCP server yang bisa dikonsumsi oleh Claude Desktop dan custom clients
  • Tahu kapan multi-agent worth the complexity dan kapan single agent dengan planning yang baik sudah cukup
  • Bisa implement trust boundary antar agent: typed contracts, confidence scoring, dan deteksi prompt injection dari subagent yang memproses external data

Production Engineering (Fase 4)

  • Bisa trace dan debug agent yang failing di production dalam waktu <30 menit menggunakan structured logs dan traces
  • Paham semua attack vectors (prompt injection, tool abuse, data leakage) dan implement defense-in-depth
  • Bisa evaluate agent performance secara kuantitatif dengan golden datasets dan LLM-as-judge — bukan hanya vibe check
  • Bisa mengoptimasi agent untuk 2x–10x lebih murah tanpa degradasi kualitas yang signifikan
  • Bisa build sandboxed execution environment yang aman untuk code execution di production

Build Production Agent (Fase 5)

  • Bisa build coding agent yang bisa handle real-world repository, run tests, dan iterate sampai selesai
  • Bisa implement computer use agent dengan proper vision-action loop dan safety guardrails
  • Tahu trade-off setiap major agent framework dan bisa justify pilihan arsitektur ke tim atau stakeholder
  • Bisa pakai VLM sebagai reasoning engine agent: encode gambar, manage token budget visual, dan tulis system prompt yang optimal untuk visual reasoning
  • Bisa wrap model vision sebagai tool yang dipanggil oleh LLM orchestrator — schema yang baik, hasil yang terstruktur, error handling yang robust
  • Bisa design dan implement vision pipeline orchestration: detect → crop → classify → caption, dengan handling paralel, conditional branching, dan confidence threshold
  • Sudah punya setidaknya satu production agent yang genuinely dipakai orang lain (bukan demo)
