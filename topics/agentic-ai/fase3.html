3.1 Orchestrator-Subagent Pattern // hierarki agent

Pattern paling umum di production multi-agent systems. Orchestrator bertugas planning, task decomposition, dan delegation ke subagents. Subagents adalah specialized agents yang fokus pada satu domain spesifik.

Kapan pakai multi-agent?

  • Task bisa diparalelisasi secara natural (riset 5 topik sekaligus)
  • Butuh specialization berbeda yang sulit digabung dalam satu agent
  • Task terlalu panjang untuk satu context window
  • Butuh independent verification — satu agent cek pekerjaan agent lain
  • Ingin isolasi risk — agent berbahaya terisolasi dari orchestrator
Pythonorchestrator_pattern.py
class OrchestratorAgent:
    """Orchestrator yang delegate ke specialized subagents."""
    def __init__(self):
        self.subagents = {
            "researcher": ResearchAgent(),    # Spesialis web search
            "coder":      CodingAgent(),       # Spesialis code gen/exec
            "writer":     WritingAgent(),      # Spesialis drafting
            "analyst":    AnalysisAgent(),     # Spesialis data analysis
        }
        self.harness = AgentHarness(AgentConfig(
            system_prompt="""You are an orchestrator.
Break complex tasks into subtasks and delegate to specialists.
Available agents: researcher, coder, writer, analyst.
Use delegate_to tool to assign tasks."""
        ))
        self.harness.register_tool(delegate_schema(), self.delegate)

    async def delegate(self, agent: str, task: str, context: str = "") -> str:
        """Tool yang dipanggil orchestrator untuk delegate ke subagent."""
        subagent = self.subagents.get(agent)
        if not subagent:
            return f"Error: unknown agent '{agent}'. Available: {list(self.subagents)}"
        full_task = f"{task}\n\nContext from orchestrator: {context}" if context else task
        result, run = await subagent.run(full_task)
        return f"[{agent} done in {run.iterations} steps]\n{result}"

    async def run(self, goal: str) -> str:
        result, _ = await self.harness.run(goal)
        return result
Jangan over-engineerMulti-agent meningkatkan kompleksitas secara eksponensial: lebih banyak LLM calls = lebih mahal, lebih lambat, lebih sulit di-debug. Mulai dengan single agent. Tambah multi-agent hanya ketika ada bottleneck yang jelas.

3.2 Parallelism & Fan-Out/Fan-In // eksekusi concurrent

Salah satu keuntungan terbesar multi-agent adalah kemampuan mengeksekusi subtasks secara paralel. Research 5 topik bisa selesai dalam waktu 1 topik jika dijalankan concurrent dengan asyncio.gather().

Pythonparallel_agents.py
import asyncio
from typing import Callable, Any

async def fan_out(
    tasks: list[str],
    agent_factory: Callable[[], Any],
    max_concurrent: int = 5
) -> list[tuple[str, str]]:
    """
    Fan-out: jalankan list tasks secara parallel.
    Fan-in: kumpulkan semua hasil.
    Returns: list of (task, result) tuples.
    """
    sem = asyncio.Semaphore(max_concurrent)   # Rate limit API calls

    async def run_one(task: str) -> tuple[str, str]:
        async with sem:
            agent = agent_factory()   # Fresh agent per task = isolated context
            result, _ = await agent.run(task)
            return task, result

    results = await asyncio.gather(
        *[run_one(t) for t in tasks],
        return_exceptions=True
    )
    successes = [(t, r) for t, r in results if not isinstance(r, Exception)]
    return successes

# topics = ["AI safety", "LLM efficiency", "multi-agent systems"]

# Speculative execution: run N approaches, take the best
async def speculative_run(task: str, n_variants: int, judge_fn) -> str:
    results = await fan_out(
        [task] * n_variants,
        agent_factory=lambda: SolvingAgent(),
        max_concurrent=n_variants
    )
    # Pick best result via judge function (could be another LLM call)
    return max(results, key=lambda x: judge_fn(x[1]))[1]

3.3 MCP — Model Context Protocol // standar universal tool exposure

MCP adalah open protocol yang awalnya dikembangkan Anthropic, namun kini telah diadopsi lintas vendor dan framework sebagai standar industri. Tujuannya: standardisasi cara LLM berinteraksi dengan external tools dan data sources — analoginya seperti USB-C untuk AI tools. Pembaca baru sering mengira MCP eksklusif Anthropic/Claude, padahal ekosistemnya sudah lintas platform (OpenAI, Google, komunitas open-source).

MCP Server

Program yang mengekspose capabilities (tools, resources, prompts) via MCP protocol. Build dengan Python atau TypeScript SDK. Contoh: MCP server untuk GitHub, PostgreSQL, Slack, filesystem.

MCP Client

Application yang terhubung ke MCP servers dan meneruskan tool calls ke LLM. Claude Desktop adalah MCP client. Kamu juga bisa build custom MCP client dalam agent-mu.

Transport Layer

MCP mendukung dua transport: stdio (local process, lebih cepat) dan SSE/HTTP (remote server, lebih fleksibel). Untuk production, biasanya pakai HTTP transport.

Resources & Prompts

Selain tools, MCP juga bisa expose resources (data yang bisa dibaca) dan prompts (template instruksi). Tiga primitif ini membuat MCP lebih kaya dari sekadar tool calling.

Pythonminimal_mcp_server.py
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent
import asyncio

app = Server("notes-mcp")

                    "limit": {"type": "integer", "default": 20}
                }
            }
        ),
        Tool(
            name="create_note",
            description="Create a new note with title, content, and tags.",
            inputSchema={
                "type": "object",
                "properties": {
                    "title": {"type": "string"},
                    "content": {"type": "string"},
                    "tags": {"type": "array", "items": {"type": "string"}}
                },
                "required": ["title", "content"]
            }
        )

@app.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "list_notes":
        notes = await db_list_notes(**arguments)
        return [TextContent(type="text", text=str(notes))]
    elif name == "create_note":
        result = await db_create_note(**arguments)
        return [TextContent(type="text", text=str(result))]

    async with stdio_server() as (r, w):
        await app.run(r, w, app.create_initialization_options())

3.4 Agent Design Patterns // pola-pola yang sudah terbukti
PatternCara kerjaUse case idealTrade-off
Router AgentClassifies input, delegates ke specialized handler yang tepatCustomer service, intent routing, triageSimple, tapi classifier harus akurat
Evaluator-OptimizerGenerator agent buat solusi → Critic agent evaluasi → revise loopCode review, essay writing, solution refinementHigher quality tapi 2x+ LLM calls
Speculative ExecutionJalankan multiple approaches parallel, ambil yang terbaikTasks dengan multiple valid solutionsExpensive tapi faster wall-clock time
Reflection / Self-CritiqueAgent evaluasi output-nya sendiri sebelum return ke userTasks butuh high accuracy, fact-checkingBetter output, tambah latency satu LLM call
Human-in-the-LoopAgent pause di checkpoints kritis, minta human inputHigh-stakes actions, ambiguous decisionsSafer tapi tidak fully autonomous
Subagent-as-ToolAgent lain di-expose sebagai tool ke orchestratorEncapsulating complex capabilityClean interface, menyembunyikan kompleksitas

Cara memilih pattern yang tepat

Evaluator-Optimizer cocok ketika kualitas sangat penting dan kamu rela bayar 2x token. Router cocok ketika kamu punya banyak intent berbeda yang perlu handler spesifik. Reflection cocok sebagai safety net untuk tasks kritis — jalankan reflection hanya untuk outputs yang akan langsung diterima user, bukan untuk setiap intermediate step.

Agent Design Patterns — Kapan menggunakan masing-masing
Router
Gunakan ketika: ada banyak intent berbeda yang butuh handler spesifik (customer service, triage)
Evaluator-Optimizer
Gunakan ketika: kualitas output sangat penting dan latency bukan constraint utama
Speculative Exec
Gunakan ketika: ada multiple valid approaches dan kamu bisa buang yang kalah
Reflection
Gunakan ketika: output langsung ke user, butuh self-check sebelum publish
Human-in-Loop
Gunakan ketika: action irreversible atau risk tinggi — deploy, delete, kirim email penting
Subagent-as-Tool
Gunakan ketika: ingin encapsulate capability kompleks dengan interface bersih ke orchestrator
3.5 Komunikasi & State Sharing // koordinasi antar agent

Multi-agent systems butuh cara untuk berbagi informasi. Ada beberapa pendekatan utama dengan trade-off yang berbeda:

Shared State Store

Semua agents baca/tulis ke shared store (Redis, DynamoDB, atau in-memory dict). Sederhana tapi bisa ada race conditions. Gunakan locks untuk operasi yang butuh atomicity.

Message Queue

Agents berkomunikasi via messages (Redis Streams, asyncio Queue, atau RabbitMQ). Lebih scalable dan decoupled. Cocok untuk event-driven architectures dengan banyak agents.

Blackboard Pattern

Shared "blackboard" di mana agents bisa post dan read partial solutions. Orchestrator memonitor blackboard dan koordinasi. Klasik untuk collaborative problem solving.

Context Passing (Recommended)

Orchestrator secara eksplisit pass context yang relevan ke setiap subagent saat delegation. Paling simple dan transparent — tidak ada implicit global state. Mulai di sini.

RekomendasiGunakan context passing sampai kamu benar-benar butuh shared state. Implicit global state membuat debugging multi-agent system jauh lebih sulit — terutama ketika race conditions muncul di production.
3.6 Agent Handoff & Trust Boundary // seberapa jauh kamu percaya subagent?

Dalam sistem multi-agent, orchestrator mendelegasikan task ke subagent dan menerima hasilnya kembali. Pertanyaan kritis yang sering diabaikan: seberapa jauh hasil dari subagent bisa dipercaya? Subagent adalah LLM — ia bisa halusinasi, salah eksekusi, atau bahkan terpapar prompt injection dari data eksternal yang ia proses.

Trust Model dalam Multi-Agent System

Percaya penuh

Subagent yang kamu control sendiri, berjalan di infrastruktur yang sama, dengan system prompt yang kamu tentukan. Hasil masih perlu divalidasi tapi trust level tinggi.

Percaya dengan verifikasi

Subagent yang memproses data eksternal (web scraping, user-provided files). Output harus divalidasi sebelum dijadikan input untuk aksi berisiko tinggi.

Tidak percaya langsung

Third-party agents, agents yang terpapar untrusted content, atau hasil dari tool yang bisa dimanipulasi. Wajib sanitasi dan validasi sebelum dipakai.

Prinsip zero-trust

Treat setiap agent output seperti input eksternal. Validasi schema, periksa konsistensi, dan jangan izinkan subagent override security constraints orchestrator.

Pythonagent_handoff.py
from pydantic import BaseModel, validator
from enum import Enum

class TrustLevel(Enum):
    HIGH   = "high"    # internal agent, controlled infra
    MEDIUM = "medium"  # internal agent, processes external data
    LOW    = "low"     # third-party agent or untrusted content

class AgentResult(BaseModel):
    agent_id: str
    task_id: str
    output: str
    confidence: float   # 0.0 – 1.0, self-reported — jangan anggap sebagai probabilitas kebenaran
    sources: list[str]  # referensi yang dipakai
    trust_level: TrustLevel

    @validator("confidence")
    def validate_confidence(cls, v):
        if not 0.0 <= v <= 1.0: raise ValueError("confidence must be 0-1")
        return v

class TrustGate:
    def validate(self, result: AgentResult, action_risk: str) -> bool:
        # LOW trust: tidak boleh trigger aksi high-risk
        if result.trust_level == TrustLevel.LOW and action_risk == "high":
            return False

        # Confidence rendah: butuh human review untuk aksi apapun
        if result.confidence < 0.7:
            return self._request_human_review(result)

        # Cek prompt injection: output tidak boleh mengandung instruksi baru
        injection_patterns = ["ignore previous", "new instructions", "system:"]
        if any(p in result.output.lower() for p in injection_patterns):
            raise SecurityError(f"Possible prompt injection in agent {result.agent_id}")

        return True

    def _request_human_review(self, result: AgentResult) -> bool:
        # Implementasi: kirim ke queue review, pause agent, tunggu approval
        print(f"[REVIEW REQUIRED] Agent {result.agent_id}: confidence={result.confidence}")
        return False  # block sampai ada approval

Pola handoff yang aman

  • Typed contracts — define schema eksplisit untuk output setiap subagent menggunakan Pydantic atau dataclass. Reject output yang tidak sesuai schema — jangan coba parse manual.
  • Confidence score — minta subagent melaporkan confidence level-nya. Output dengan confidence rendah harus di-flag untuk human review. Penting: self-reported confidence dari LLM tidak berkorelasi langsung dengan kebenaran — lebih dipercaya adalah verification result, evaluator model, atau tool success rate. Gunakan confidence sebagai sinyal, bukan ground truth.
  • Source tracking — setiap klaim dalam output subagent harus traceable ke sumber. Ini krusial untuk debugging ketika ada rantai agent yang panjang.
  • Blast radius limiting — subagent tidak boleh memiliki akses yang lebih luas dari yang dibutuhkan untuk task-nya. Coding subagent tidak perlu akses ke database production.
Prompt injection via subagentJika subagent memproses konten dari web atau user-provided files, ia bisa terpapar "prompt injection" — instruksi berbahaya yang tertanam dalam konten eksternal yang mencoba mengubah perilaku agent. Selalu sanitasi output subagent yang memproses untrusted data sebelum dipakai sebagai input untuk aksi berisiko.
3.7 Agent-to-Agent Protocol & Negotiation // dua agent bicara satu bahasa

Saat dua agent perlu berkolaborasi, mereka butuh protokol yang jelas — bukan hanya "kirim pesan", tapi struktur yang memungkinkan satu agent memahami kapabilitas agent lain, memformat request dengan benar, dan menangani response yang tidak terduga. Tanpa ini, multi-agent system menjadi chaos yang sulit di-debug.

Empat elemen protokol antar agent

  • Capability advertisement — setiap agent mengekspos apa yang bisa dilakukannya dalam format terstandar (mirip OpenAPI spec untuk agent)
  • Structured request format — bukan plain text, tapi JSON schema yang ketat: task, context, constraints, expected output format
  • Response contract — agent yang dipanggil wajib return dalam format yang disepakati: success/failure, result, confidence, reasoning
  • Rejection handling — agent yang dipanggil bisa menolak task (out of scope, overloaded, conflict of interest) dan orchestrator harus handle ini

# Orchestrator query capability sebelum assign task
def can_handle(agent_manifest: dict, task_type: str) -> bool:
    return any(c["name"] == task_type for c in agent_manifest["capabilities"])
Protocol over improvisationAgent yang berkomunikasi lewat plain text rentan terhadap misinterpretasi dan prompt injection antar agent. Schema yang ketat memaksa struktur dan memudahkan validasi — orchestrator bisa assert bahwa response valid sebelum menggunakannya untuk langkah berikutnya.
